## [Go away Python](https://lorentz.app/blog-item.html?id=go-shebang)
**Score:** 297 | **Comments:** 288 | **ID:** 46431028

> **Article:** The article "Go away Python" proposes using Go for shell scripting as an alternative to Python. The author presents a clever hack using a Go source file as its own launcher. By starting the file with a specific multi-line comment (`///usr/bin/env go run "$0" "$@"; exit`), it can be executed directly as a script. The first line is interpreted by the shell as a shebang, while the subsequent lines are treated as a comment by the Go compiler. This allows developers to write and run Go scripts without a separate build step, aiming to combine Go's performance and type safety with the convenience of scripting.
>
> **Discussion:** The Hacker News discussion reveals a split between those who appreciate the cleverness of the hack and those who question its practicality for scripting. The core debate centers on "language ergonomics" and the right tool for the job. One commenter, PaulRobinson, articulated a popular sentiment: Go feels designed for "shippable" software that encourages best practices like testing and formal pipelines, whereas scripting thrives on "quick, dirty, throwaway" code, a niche where languages like Python and bash excel. This idea was supported by others who shared experiences of Python's dependency and environment complexities causing friction, while Go's compile-time guarantees offer reliability.

Several practical improvements and alternatives were suggested:
*   **Correctness:** A user noted that the script should use `exit $?` to pass the exit code from the Go program back to the shell.
*   **Portability:** Another suggested using `///usr/bin/env go run...` for better cross-platform compatibility.
*   **Modern Python:** A major counterpoint was raised that the Python ecosystem tool `uv` already solves the author's complaints about dependency management. `uv` can create self-contained executable scripts that manage their own Python version and PyPI dependencies via a simple shebang, making Python highly viable for this use case.
*   **Other Languages:** Users also pointed out that C# (.NET) has official support for file-based programs with shebangs, and even Java has similar capabilities.

Ultimately, while the Go hack was seen as "mad genius stuff," the broader conversation questioned whether Go's culture and tooling are a natural fit for the scripting paradigm, with many arguing that modern Python tooling or other languages might be better suited.

## [Win32 is the stable Linux ABI](https://loss32.org/)
**Score:** 164 | **Comments:** 1 | **ID:** 46433035

> **Article:** The article "Win32 is the stable Linux ABI" argues that the most stable and reliable Application Binary Interface (ABI) for running Windows software on Linux is not a translation layer like Wine, but the actual Windows kernel itself, running inside a virtual machine. The author posits that the Linux kernel's internal ABI is constantly changing and unstable, making it a poor target for binary compatibility. In contrast, the Win32 API is a stable, decades-long contract with developers.

The proposed solution is a lightweight, "micro" Windows virtual machine that runs the Windows kernel but is deeply integrated with the host Linux system. This VM would share the host's filesystem, networking stack, and potentially even the display server (Wayland). The goal is to create a seamless experience where Windows applications feel native to the Linux desktop, but their underlying system calls are handled by the stable, predictable Windows kernel inside the VM, thus achieving superior compatibility and stability than any other method.
>
> **Discussion:** The Hacker News discussion is highly engaged, with users exploring the technical merits, feasibility, and broader implications of the idea. The core debate centers on whether this approach is a brilliant solution to a long-standing problem or an overly complex workaround.

A significant portion of the conversation focuses on the technical challenges and potential of the concept. Many commenters point out that this is not a new idea, referencing projects like User Mode Linux (UML), QEMU's microvm, and the long-standing goal of virtualization without the overhead. The discussion delves into the complexity of achieving deep integration, particularly with graphics (GPU drivers) and input systems. Some users suggest that a paravirtualized approach, where the guest OS is aware it's running on a hypervisor, might be more practical than full emulation. There's also a debate about the performance overhead of context switching between the host and guest kernels, with some arguing it would be negligible for most applications while others see it as a significant hurdle.

The discussion also explores the philosophical and strategic dimensions of the proposal. A key theme is the contrast between the Linux kernel's rapid evolution and the Windows API's stability. Many agree with the article's premise that the Linux kernel ABI is a moving target, which is a feature for innovation but a bug for binary compatibility. The conversation then splits on the solution. Some see this as a pragmatic acceptance that the Win32 ecosystem is too large to ignore and that "embrace, extend, extinguish" could be flipped to "embrace, extend, enhance" the open-source desktop. Others argue that this approach cedes too much ground to Microsoft and that the focus should remain on improving Wine and native Linux development. The idea is compared to Apple's successful Rosetta 2, which also uses a translation/virtualization layer to provide seamless compatibility.
