- Male mice exposed to the scent of cherry blossoms or the drug acetophenone passed down a heightened sensitivity to that scent to their offspring and even grandchildren.

## [C++ says “We have try. . . finally at home”](https://devblogs.microsoft.com/oldnewthing/20251222-00/?p=111890)
**Score:** 106 | **Comments:** 121 | **ID:** 46408984

> **Article:** The article, from Raymond Chen's "The Old New Thing" blog, uses a popular meme format ("We have X at home") to explain a C++ concept. The title "C++ says 'We have try...finally at home'" implies that C++ already has a mechanism equivalent to the `try...finally` block found in languages like Java or C#. This mechanism is **RAII (Resource Acquisition Is Initialization)**, specifically implemented through C++ **destructors**. The core idea is that an object's destructor is automatically called when it goes out of scope, guaranteeing cleanup code is executed regardless of how the scope is exited (e.g., via return, exception, or normal execution), thus providing the same function as a `finally` block.
>
> **Discussion:** The Hacker News discussion revolves around the comparison between C++'s RAII/destructor model and the explicit `try...finally` construct. Key points include:

*   **RAII vs. `finally`:** Many commenters argue that RAII is superior because it's less verbose, avoids nested blocks, and ensures cleanup is tied to an object's lifetime, reducing the chance of forgetting a `finally` block. However, others point out that RAII requires creating a class, which can be more cumbersome than an inline `finally` block for simple cases.
*   **Alternative Approaches:** Users brought up `defer` blocks, as seen in Swift and Go, as a more universal and convenient solution for scope-based cleanup that isn't tied to exception handling. A C++ macro implementation for `defer` was also shared.
*   **Error Handling Nuances:** A significant sub-thread focused on the pitfalls of both models. One commenter criticized how many languages (Java, Python, JS) handle exceptions thrown in `finally` blocks, arguing they obscure the original, more important error. Another pointed out that C++'s behavior of crashing if a destructor throws is even worse.
*   **Readability and Complexity:** The discussion touched on C++'s notorious syntax complexity, with some users finding it unreadable while others have become accustomed to it.
*   **RAII Limitations:** A counterpoint was raised that destructors closing files can be problematic if the close operation itself fails (e.g., on networked file systems), as the error is difficult to handle.
