## [Go away Python](https://lorentz.app/blog-item.html?id=go-shebang)
**Score:** 365 | **Comments:** 334 | **ID:** 46431028

> **Article:** The article "Go away Python" proposes using Go for shell scripting via a clever shebang hack. The method involves starting a Go source file with a comment that is also a shell command: `///usr/bin/env go run "$0" "$@"; exit`. The author argues this is a viable alternative to Python for scripting, citing Go's static typing, performance, and single-file deployment model. The post is presented as a "troll" that the author has come to believe is a genuinely workable idea, aiming to bypass the complexities of Python's virtual environments and dependency management.
>
> **Discussion:** The discussion reveals a split between those who appreciate the cleverness of the hack and those who question its practicality. The core debate centers on the ergonomics of scripting versus building. One user eloquently argues that scripting languages like Bash and Python are designed for "quick, dirty, throwaway code," whereas Go's culture and tooling encourage a more formal, shippable software mindset with tests and build pipelines. This sentiment is echoed by others who find Go's verbosity and strictness at odds with the fluid nature of scripting.

Several technical points and alternatives were raised:
*   **Python Tooling:** A major counter-argument is that modern Python tools, specifically `uv`, have already solved the dependency and shebang issues the article critiques. Users demonstrated shebangs like `#!/usr/bin/env -S uv run --script` that manage virtual environments and dependencies seamlessly, even installing the correct Python version if needed.
*   **Exit Codes:** A crucial flaw in the proposed hack is that it doesn't pass the exit code from the Go program back to the shell. The suggested fix is to use `exit $?`.
*   **Portability:** The shebang's portability was questioned, with a more portable version suggested (`///usr/bin/env go run "$0" "$@"; exit`).
*   **Other Languages:** Users noted similar capabilities in C (with `tcc`), C# (.NET), and Java, indicating this is not a new concept.

Overall, while the hack was seen as "mad genius," the consensus leaned towards it being a novelty rather than a practical replacement for established scripting languages or modern Python tooling.

The conversation explored how more advanced protocols can safely manage the curve's non-prime-order nature or use abstractions like Ristretto255 to provide a prime-order group, which was praised by one user for its elegant API.

The piece highlights that projects like Wine and Valve's Proton allow Linux applications (and Windows applications running on Linux) to rely on a consistent set of Win32 APIs that do not break with kernel updates. >
> **Discussion:** The discussion largely validates the article's central thesis, with many commenters providing historical and technical context. A primary theme is the distinction between the Linux kernel's internal ABI and its user-space system call (syscall) ABI. While the syscall ABI is generally stable, the article's argument resonates because the kernel community's development philosophy prioritizes internal refactoring and correctness over preserving the stability of internal interfaces, which can break out-of-tree modules (like proprietary drivers). This has led to frequent breakage for projects like NVIDIA's driver, which has had to adapt to kernel changes repeatedly.

Commenters point to Valve's massive investment in Proton and Wine as the real-world proof of this concept. By abstracting games behind the Win32 API, Valve ensures that a huge library of software continues to run on Linux without needing constant updates from developers. This is contrasted with the native Linux desktop, where the "stable ABI" is a constantly shifting landscape of Wayland, X11, and various other APIs.

There is some debate on the semantics. Some argue that the Linux syscall ABI *is* stable and that the article conflates it with internal kernel APIs. However, the prevailing sentiment is that for practical, long-term software development, the Win32 layer provides a more predictable and less volatile environment. The discussion concludes that the Win32 ABI's stability, when implemented by a compatibility layer like Wine, is a pragmatic and powerful feature for Linux, even if it's an ironic one.
