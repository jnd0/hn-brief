## [Win32 is the stable Linux ABI](https://loss32.org/)
**Score:** 165 | **Comments:** 1 | **ID:** 46433035

> **Article:** The article "Win32 is the stable Linux ABI" argues that the Win32 API has effectively become a more stable and reliable Application Binary Interface (ABI) for Linux than the native Linux kernel ABI itself. The author contends that Microsoft, driven by the need for long-term enterprise support and backward compatibility for Windows applications, has been forced to maintain an extremely high level of stability for the Win32 API over decades. In contrast, the Linux kernel developers, prioritizing performance, rapid innovation, and internal refactoring, frequently change internal kernel structures and system call interfaces. This forces Linux-native applications to be constantly recompiled to keep up with kernel updates. The author posits that running Windows applications via compatibility layers like Wine or Proton provides a more stable target for developers, as the Win32 API is unlikely to break, whereas native Linux development is a moving target.
>
> **Discussion:** The discussion on Hacker News is highly polarized, with users debating the core premise and the definitions of "ABI" and "stability."

A significant portion of the comments strongly disagree with the article's thesis. These detractors argue that the author conflates the stable, public Win32 *API* with a low-level *ABI*. They point out that while the Win32 API surface is stable, the underlying Windows kernel and its ABI are also subject to changes, and that Microsoft has made significant, breaking changes over the years (e.g., the transition from 32-bit to 64-bit, or security patches that alter behavior). They also highlight that the Linux kernel ABI for user-space is, in fact, remarkably stable; system calls are rarely removed or changed in a breaking way, and applications compiled years ago often still run. This camp suggests the article's argument is based on a flawed comparison.

Conversely, a group of commenters defends the article's core sentiment, arguing that it holds a "kernel of truth." They focus on the user and developer experience, agreeing that the *promise* of Win32 stability is a powerful draw. For a developer targeting Windows, the API is a fixed, documented target. For Linux, they argue, the ecosystem is fragmented, with different libraries, display servers (X11 vs. Wayland), and package formats creating instability from the developer's perspective, even if the kernel syscalls themselves are stable. They see the Win32 ABI as a "stable by decree" model from a single vendor, which contrasts with the "stable by convention" and more chaotic nature of the open-source Linux world.

A third, more nuanced thread of discussion emerges around the practical implications for software development. Many commenters use this as a springboard to discuss the challenges of distributing software on Linux. They argue that the lack of a single, stable, cross-distro binary format and API (like Win32/PE for Windows or Mach-O for macOS) is a major weakness. This leads to the "dependency hell" problem, where an application compiled on one distro may not run on another due to differing library versions.
