## [Go away Python](https://lorentz.app/blog-item.html?id=go-shebang)
**Score:** 320 | **Comments:** 316 | **ID:** 46431028

> **Article:** The article "Go away Python" proposes using Go for shell scripting via a clever shebang hack. The author demonstrates a script that begins with a Go comment containing a shell command: `///usr/bin/env go run "$0" "$@"; exit`. The shell executes this line, running the script itself with `go run`. The Go compiler then parses the file, but because Go comments start with `//`, the first line is ignored as a comment, allowing the script to run without modification. The author argues this makes Go a viable alternative to Python for scripting, citing benefits like single-file deployment and static typing, while dismissing the lack of an interpreter mode as a non-issue with this method.
>
> **Discussion:** The Hacker News discussion largely treated the article as a clever but impractical hack, focusing on the ergonomics of scripting versus compiled languages. The consensus was that while the technique works, Go's workflow is fundamentally unsuited for the "quick and dirty" nature of scripting.

Key themes included:
*   **Language Ergonomics:** Commenters distinguished between "scripting" languages (bash, Python, Ruby) and "shippable" languages (Go, Rust). The latter encourage formal tooling, testing, and build pipelines, which are antithetical to the throwaway nature of many scripts. The verbosity and strictness of Go were cited as friction points.
*   **Python Tooling Evolution:** Several users pushed back on the article's dismissal of Python's ecosystem. They highlighted that modern tools like `uv` have solved Python's dependency and environment management issues for scripting, offering a shebang-based workflow that handles dependencies and even Python versioning automatically.
*   **Alternatives and Comparisons:** Users noted that similar shebang hacks exist for C and TCC. Others pointed out that .NET/C# has official, first-class support for file-based programs and shebangs, making it a more robust alternative for this use case.
*   **Technical Nuances:** The discussion included minor technical critiques, such as the need to pass the correct exit code back to the shell and ensuring portability by using `env`.

The motivation is to eliminate a common source of "AI slop" vulnerability reports. Sternberg notes that `strcpy` acts as a "honeypot" for AI-powered code scanners, which frequently generate false-positive reports by inventing complex, but logically flawed, proofs of vulnerability.

To replace `strcpy`, the team introduced a custom function, `curlx_strcopy`. This function requires the destination buffer size and the source string length, performing a runtime check to ensure the copy (including the null terminator) will fit. If the check fails, the function triggers a debug assertion and sets the destination string to empty. The article concludes that removing `strcpy` is a pragmatic way to reduce the noise from low-quality AI-generated security reports.
>
> **Discussion:** The discussion centered on the effectiveness of the proposed solution, the broader problem of AI-generated vulnerability reports, and the nuances of C string handling.

A significant portion of the conversation focused on the AI "slop" problem. Commenters noted that while some AI-powered analyzers are genuinely useful and have found hundreds of real bugs, many reports are low-effort attempts to collect bounties. These reports don't just flag a function call but invent a flawed "proof" of a vulnerability, making them time-consuming to debunk. Some users even suggested creating "honeypot" functions to waste the time of these automated reporters.

There was considerable debate about the `curlx_strcopy` replacement. While some praised the goal of coupling size checks with the copy operation, several commenters criticized its design. Key criticisms included its `void` return type (which doesn't explicitly signal failure to the caller) and its behavior of setting the destination to an empty string on failure instead of leaving it untouched. Some argued this was a poor trade-off and that using `memcpy` directly or a function with an explicit error code would be better.

Finally, the discussion branched into a general debate on C string functions. Commenters discussed the correct use of `strncpy` (for fixed-width fields, not general safety), the limitations of various standard library functions, and the need for string libraries that track buffer capacity alongside the pointer.

A "shim" driver within the VM would intercept system calls from Windows applications and translate them into direct, high-performance requests to the host Linux kernel for hardware resources (like graphics and input). This approach aims to provide the rock-solid compatibility of a full Windows environment while achieving near-native performance by bypassing traditional virtualization overhead and directly leveraging the host Linux system. Commenters with deep kernel development experience argued that intercepting and translating all the necessary kernel calls (not just filesystem or network, but also graphics, audio, input, and the myriad of undocumented Windows kernel APIs) is a monumental, if not impossible, task.
