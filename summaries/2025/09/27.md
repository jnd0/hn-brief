## [SSH3: Faster and rich secure shell using HTTP/3](https://github.com/francoismichel/ssh3)
**Score:** 532 | **Comments:** 263 | **ID:** 45395991

> **Article:** The article links to a GitHub repository for "SSH3," a project that implements a secure shell protocol using HTTP/3 and QUIC. The goal is to provide a faster connection setup time compared to traditional SSH by leveraging the reduced round trips of HTTP/3. It also aims to solve other perceived shortcomings of SSH, such as the "Trust on First Use" (TOFU) model for host verification, by using standard TLS certificates (like Let's Encrypt). The project also notes that it can work through standard HTTP/S proxies and firewalls, which often block traditional SSH ports.
>
> **Discussion:** The Hacker News discussion is largely skeptical and analytical. Key points include:

*   **Performance Clarification:** Users clarify that the primary speed benefit is in connection establishment, not in the throughput of an active session, which aligns with the nature of HTTP/3. It's also noted that UDP-based transport could be significantly better for high-latency links.
*   **"Everything over HTTP":** A recurring theme is the feeling of sadness or skepticism that all application-layer protocols are being absorbed into HTTP. Some argue this is a natural evolution as the underlying transport (QUIC) becomes a general-purpose alternative to TCP, while others feel it's a "kludge" to keep calling it HTTP.
*   **Project Viability:** Commenters point out the project appears to be inactive (no recent commits), labeling it more of a proof-of-concept than a serious contender to replace OpenSSH.
*   **Security Concerns:** One user raises the concern that running an SSH-like service over HTTP would expose it to a much larger volume of automated bot scans and potential HTTP exploits.
*   **Naming:** The name "SSH3" is widely disliked as confusing and misleading, with users noting the project itself plans to rename it.
*   **Use Cases:** A practical use case mentioned is bypassing corporate firewalls that block outbound SSH but allow HTTPS traffic.
*   **"If it ain't broke...":** A debate on whether SSH is "broken." Proponents of SSH3 argue that SSH's host key verification (TOFU) and authentication models are clunky compared to modern web standards like TLS certificates and OAuth, while others defend SSH's established utility.

## [Users only care about 20% of your application](https://idiallo.com/blog/users-only-care-about-20-percent)
**Score:** 405 | **Comments:** 189 | **ID:** 45395468

> **Article:** The article, which appears to be a blog post on idiallo.com, discusses the Pareto principle (80/20 rule) in the context of software applications. It argues that users typically only use about 20% of an application's features. However, the author's main point is that this isn't a flaw. The "other 80%" is not necessarily bloat, because different users utilize different 20% slices. The application as a whole serves a diverse user base, each with their own specific needs. The article advises developers to embrace this "pluralism" and focus on interoperability, allowing users to solve their problems in various ways rather than trying to force a one-size-fits-all "simple" interface.
>
> **Discussion:** The HN discussion largely validates the article's premise, with many commenters pointing out that the concept is not new and is well-articulated in classic software development literature, specifically by Joel Spolsky (who is credited in several comments). The key points of the discussion are:

*   **The 80/20 Rule is an Understatement:** Several users argue that for complex software like Microsoft Word, the percentage of features used by the average person is far lower than 20%, perhaps closer to 1-2%.
*   **User Motivation is Key:** Users don't engage with features they don't have a problem to solve for. They are not just "users" but people with specific tasks, and they will ignore features that don't relate to their immediate needs.
*   **The "Different 20%" Insight:** A highly upvoted comment reiterates the article's main thesis: the 20% is not a fixed set of features but varies from user to user.
*   **Developer Strategy:** The discussion touches on how developers should respond. One suggestion is to focus on "paving the bare spots" or following "desire paths" (observing how users actually use the software and improving those workflows) rather than trying to predict all use cases.
*   **Interoperability:** A comment highlights that the best way to handle diverse needs is to make the application work well with other tools, embracing the "Unix philosophy" of doing one thing well and chaining tools together.

>
> **Discussion:** The Hacker News discussion centered on several key themes:

The core issue is trusting code from unknown authors.
*   **The "Wild West" of MCP:** While the attack vector is not new, participants noted that the MCP ecosystem is particularly vulnerable because it's so new. Unlike established libraries with long-standing reputations, MCP servers lack a history of community vetting, making it easier for malicious packages to gain traction.
*   **Comparison to Big Tech:** Some users drew parallels to the data collection practices of major corporations like Microsoft and Google, questioning why a single developer's backdoor is seen as a bigger deal than systemic data harvesting by large companies. The consensus was that malicious intent from an unknown attacker is different from the known (though often undesirable) policies of a large corporation.
*   **Skepticism and Developer Responsibility:** A recurring sentiment was that developers should be more suspicious of dependencies. Some advocated for using simpler, more direct methods (like `curl` or direct API calls) instead of installing bloated or unaudited SDKs. There was a general agreement that installing packages from strangers is a significant liability.
*   **User Awareness and "Common Knowledge":** A sub-thread debated whether the average user is oblivious to these risks.
