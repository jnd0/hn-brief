## [We found a bug in Go's ARM64 compiler](https://blog.cloudflare.com/how-we-found-a-bug-in-gos-arm64-compiler/)
**Score:** 835 | **Comments:** 140 | **ID:** 45516000

> **Article:** Cloudflare engineers discovered and diagnosed a subtle bug in the Go compiler for the ARM64 architecture. The issue occurs when a function's stack frame is large enough to require a stack pointer adjustment greater than what can be represented in a single 32-bit immediate instruction. The compiler, in an attempt to perform this large adjustment, splits it into two separate instructions (e.g., two `ADD` operations). This creates a critical window where the stack pointer is in an inconsistent state. If an asynchronous event (like a goroutine preemption or profiling signal) occurs between these two instructions, the system can attempt to unwind the stack using a corrupt state, leading to a crash. The article details their detective work, which involved writing custom assembly to reproduce the issue and ultimately pinpointing the non-atomic stack adjustment as the root cause. The fix involves ensuring the compiler generates a sequence that keeps the stack pointer valid at all times, even if it costs an extra instruction.
>
> **Discussion:** The Hacker News discussion is largely a mix of praise for the technical write-up and insightful analysis of the bug's nature. The consensus is that this was a classic, hard-to-find concurrency bug that only manifests at scale.

Key insights from the discussion include:
*   **The Root Cause:** Commenters quickly grasped that the problem was the non-atomic stack pointer adjustment. As `mperham` clarifies, the frame pointer was also corrupted in this window, making unwinding impossible. The bug is a direct result of the compiler splitting a single logical operation into multiple machine instructions.
*   **The "Why":** Several users speculated on the origin. `whizzter` suggested it was an artifact of x86-centric thinking, where large immediates can often be encoded in a single instruction, leading ARM-specific edge cases to be overlooked during development.
*   **The Fix:** A proposed fix, which aligns with the eventual upstream patch, was to load the large constant into a register first and then perform a single atomic `ADD` to the stack pointer, trading an extra instruction for correctness.
*   **Broader Takeaways:** The community was split on the lesson. Some, like `gok`, argued it was a cautionary tale against "rolling your own" toolchain components (like Go's assembler), suggesting reliance on established tools like LLVM would be safer. Others countered that implementing a language's backend is a necessary part of the job.
*   **Cloudflare Context:** A side discussion noted Cloudflare's significant and growing use of ARM64 in production, with users linking to previous blog posts about their server infrastructure.

There were no significant disagreements; the tone was one of appreciation for a well-executed engineering investigation.
