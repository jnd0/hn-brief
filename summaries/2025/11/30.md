In essence, the discussion reveals that `CLAUDE.md` is a form of explicit context management, a necessary piece of scaffolding for current-generation coding agents that lack persistent memory. ## [Windows drive letters are not limited to A-Z](https://www.ryanliptak.com/blog/windows-drive-letters-are-not-limited-to-a-z/)
**Score:** 502 | **Comments:** 281 | **ID:** 46096556

> **Article:** The article reveals that Windows drive letters are not technically restricted to the A-Z range, despite being the only ones exposed in the File Explorer GUI. The underlying NTFS object manager allows for much longer and more complex "drive" names (like `€:\` or `FOO:\`) to be created and used via the Win32 API. This works because Windows translates DOS-style paths (e.g., `C:\foo`) into NT object paths (e.g., `\??\C:\foo`). The article demonstrates that by creating an NTFS object named `C:` (or any other string), that string can effectively function as a drive letter. This legacy abstraction is a remnant of the NT kernel's design, which is far more flexible than the DOS-compatible façade presented to most users.
>
> **Discussion:** The Hacker News discussion is a mix of nostalgia, technical appreciation, and security concerns. The general consensus is that this is a "cool but cursed" piece of trivia that highlights the often-hidden flexibility and legacy cruft of the Windows NT kernel.

Key insights from the discussion include:
*   **Historical Context:** Commenters reminisce about the days when A: and B: were for floppy drives and C: was a luxury. There's also a notable mention that the original Xbox 360 used this flexibility for its own file system, with drive-like prefixes such as `Game:\` and `Hdd0:\`.
*   **Under-the-Hood Architecture:** A highly upvoted comment explains that this behavior is part of a larger, "wild" system based on reverse-mapping GUIDs and symbolic links. This architecture, while powerful, is also cited as a primary reason for Windows' historical vulnerability to malware.
*   **Security Implications:** There is debate about the malware potential. While some fear it could be used to create "annoying" or hidden drives, others point out that creating these mounts requires administrative privileges, mitigating the risk for casual exploits.
*   **Practical Workarounds:** The discussion clarifies that users aren't stuck with drive letters. A commenter points out that partitions can be mounted to directories (e.g., `C:\MyData`) using PowerShell or the Disk Management GUI, a feature that works for NTFS partitions and is a robust solution to the "drive letter theft" problem mentioned by another user.
*   **Limitations:** It's noted that these non-standard drive letters won't appear in File Explorer, limiting their utility. Furthermore, using them in command-line operations can be tricky, as some characters (like `+`) are also command-line operators.

Overall, the community treated this as a fascinating look "under the hood" of an OS they use daily, appreciating the technical depth while acknowledging its bizarre and potentially hazardous nature.
