## [Why is the first C++ (m)allocation always 72 KB?](https://joelsiks.com/posts/cpp-emergency-pool-72kb-allocation/)
**Score:** 121 | **Comments:** 27 | **ID:** 47205129

> **Article:** The article investigates why the first memory allocation in C++ programs using GCC's libstdc++ is consistently 72 KB. The author discovered this occurs due to an "emergency pool" that libstdc++ allocates during initialization to handle exceptions when the system's malloc function runs out of memory. This emergency pool is created before main() executes, which explains why it appears as the first allocation. The author demonstrates how to override malloc using LD_PRELOAD to observe this behavior and notes that the pool size can be configured or disabled through compiler flags and environment variables.
>
> **Discussion:** The discussion centered around the broader implications of the emergency pool allocation and the ease of overriding fundamental functions like malloc. Several commenters noted that this phenomenon applies to many areas of programming where people are often intimidated by perceived complexity and default to using libraries without understanding alternatives. The thread explored various methods for overriding malloc, including weak symbol binding in ELF objects and the now-deprecated malloc hooks in glibc. Some participants questioned the article's title for implying this behavior is universal rather than compiler-specific, while others debated whether static allocation of the emergency pool would be preferable. The conversation also touched on the trade-offs between dynamic and static memory allocation, with some arguing that any waste is problematic, especially in embedded environments, while others defended the approach as part of the "pay only for what you use" philosophy.
