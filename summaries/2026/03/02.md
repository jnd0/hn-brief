## [Why is the first C++ (m)allocation always 72 KB?](https://joelsiks.com/posts/cpp-emergency-pool-72kb-allocation/)
**Score:** 119 | **Comments:** 26 | **ID:** 47205129

> **Article:** The article explains that libstdc++ reserves a 72 KB “emergency pool” on the first C++ allocation to guarantee that exception handling can succeed even if the normal heap is exhausted. This pool is allocated lazily at startup via a custom allocator (mymalloc) and is used for throwing std::bad_alloc and other exception objects. The author notes that the pool size can be changed by configuring libstdc++ with --enable-libstdcxx-static-eh-pool or disabled entirely with the environment variable GLIBCXX_TUNABLES=glibcxx.eh_pool.obj_count=0. The post also links to a previous master’s thesis implementation (jsmalloc) that provides similar debug‑logging behavior.
>
> **Discussion:** Commenters praised the article for demystifying a seemingly complex part of the C++ runtime, noting that replacing malloc for simple tools like ls is surprisingly straightforward. A debate emerged over the portability of such tricks, with some emphasizing that the 72 KB pool is GCC‑specific and not a universal C++ feature, while others argued that custom memory implementations (e.g., ClickHouse’s memcpy) can be viable when properly benchmarked. Several participants shared techniques for overriding malloc on Linux, discussing weak symbols, dlsym with RTLD_NEXT, and the deprecation of glibc’s malloc hooks, highlighting both utility and thread‑safety concerns. The author clarified that a static pool can be enabled via a build flag and that the article’s title was later edited to reflect its environment‑specific nature, prompting further discussion about the trade‑offs of static versus dynamic emergency buffers. Overall, the thread blended technical insights with broader reflections on library reliance and the balance between simplicity and performance.
