## [The occasional ECONNRESET](https://movq.de/blog/postings/2026-05-05/1/POSTING-en.html)
**Score:** 114 | **Comments:** 25 | **ID:** 48170799

> **Article:** The post discusses how modern Linux TCP stacks send a reset (RST) instead of a finish (FIN) when data is lost, following RFC 2525 §2.17, and illustrates the issue with an FTP example that shows a bulk GET being suspended, a zero‑window advertisement, and a subsequent kill‑9 causing the server to hang. It explains that sending a FIN in this scenario can leave the connection in a dead‑locked state, with the server stuck on `send()` and the client waiting in FIN‑WAIT‑2 for a FIN that will never arrive. The article highlights that RST is preferred because it immediately aborts the connection, preventing long‑lived, stalled sockets that waste resources. It also notes that the difference between RST and FIN manifests in application reads as `ECONNRESET` versus end‑of‑file, which matters for protocols that rely on clean termination.
>
> **Discussion:** Debate erupted over the practical consequences of using FIN versus RST, with contributors pointing out that a FIN can leave a server thread blocked on `send()` for minutes, especially in legacy single‑threaded FTP servers, while an RST forces an immediate teardown that avoids lingering sockets. Some participants emphasized that the distinction matters at the application level, as `ECONNRESET` signals an abrupt abort whereas EOF can be interpreted as a legitimate end of data, potentially leading to protocol misbehaviour if the difference is ignored. Others argued that modern servers mitigate the issue with shutdown or lingering‑close mechanisms, yet noted that many implementations still fail to read all pending data before closing, causing resets and lost data. The discussion also touched on HTTP specifics, such as the need for `Content‑Length` or chunked encoding when closing connections, and the role of `Expect: 100‑continue` in preventing incomplete responses. Overall, the thread underscored that while RST is technically justified, careful socket handling and protocol design remain essential to avoid the “awful effects” of FIN‑based termination.
