Microsoft maintained both variables for backward compatibility with different software ecosystems - TMP was used by CP/M-86 and early DOS programs while TEMP was introduced later as the more standardized variable name. The author notes that this duplication persists today primarily for legacy application support, with modern Windows versions treating both variables equivalently. The piece illustrates how early computing constraints and competing standards created lasting technical debt in modern operating systems.
>
> **Discussion:** Commenters expanded on the historical context, with several recalling how CP/M programs were configured through binary patching rather than external configuration files due to severe memory and storage constraints - programs often ran in as little as 32K RAM with 130K floppy disks. The conversation shifted to modern configuration management, where users debated the merits of XDG Base Directory Specification adoption versus the proliferation of dotfiles across home directories, with some noting that even major projects like Firefox have been slow to comply. Environment variable practices sparked discussion about naming conventions and conflicts, with users sharing strategies like lowercase prefixes or MY_ prefixes to avoid collisions with system variables like TZ. The thread also touched on the suckless software philosophy of source-code configuration, with mixed reactions about whether manual patching was a reasonable approach or unnecessarily elitist, while some participants corrected historical timeline inaccuracies about CP/M development dates.
