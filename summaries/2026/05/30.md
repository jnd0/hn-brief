## [Bijou64: A variable-length integer encoding](https://www.inkandswitch.com/tangents/bijou64/)
**Score:** 235 | **Comments:** 82 | **ID:** 48323992

> **Article:** The article presents Bijou64, a variable-length integer encoding that differs from LEB128 by using a counter-based approach in the first byte rather than continuation bits. Values 0x0 to 0xF7 encode as single bytes, while 0xF8 to 0xFF use their upper 5 bits as a counter indicating how many subsequent bytes follow, supporting the full uint64 range without requiring a 10th byte. This design ensures each number has exactly one canonical representation, avoiding the multiple encoding problem that affects LEB128. The encoding prioritizes predictable decoding performance and security through canonicality over maximum space efficiency.
>
> **Discussion:** Commenters debated the trade-offs between Bijou64 and existing variable-length integer encodings like LEB128, with several key technical insights emerging. Some users questioned why LEB128 allows non-canonical encodings, and others explained practical use cases like reserving space for length prefixes when the payload size isn't known upfront, or using widest encodings during compilation to avoid code movement during linking. Performance considerations sparked disagreement, as kstenerud argued that Bijou64's approach breaks down with SIMD optimization compared to ULEB128, while nine_k countered that these serve different use cases - SIMD for CPU processing versus storage/transmission optimization. Security implications were a major theme, with matja and others emphasizing that canonicality prevents signature forgery and malleability attacks, though dzaima challenged whether Bijou64 actually solves this problem better than LEB128. stebalien noted that while Bijou64 offers cleaner semantics for most use cases, it sacrifices encoding density compared to LEB128, which provides 2-byte representations for values up to 16,383 versus Bijou64's ~500 two-byte numbers.
