## [Bijou64: A variable-length integer encoding](https://www.inkandswitch.com/tangents/bijou64/)
**Score:** 210 | **Comments:** 74 | **ID:** 48323992

> **Article:** The article presents Bijou64, a variable-length integer encoding scheme that uses the first byte to indicate both the value and the length of the integer. Values from 0x0 to 0xF7 are encoded as single bytes, while values 0xF8 to 0xFF use the upper 5 bits as a counter specifying how many additional bytes follow, limiting the maximum representable value to just under 2^264. The encoding ensures canonical representation where each number has exactly one valid encoding, and the length is immediately apparent from the first byte without needing to scan continuation bits. This design aims to address issues with LEB128's non-canonical encodings while maintaining compact representation for small values.
>
> **Discussion:** The discussion centered on the practical tradeoffs of variable-length integer encodings, particularly why LEB128 permits non-canonical representations despite their security risks. Several commenters explained that padding bytes in LEB128 serve legitimate purposes, such as allowing encoders to reserve space when the payload length isn't known upfront during serialization, or enabling linker-friendly code generation where symbol addresses can be patched without moving surrounding code. However, dzaima argued that bijou64 fails to actually solve the canonicality problem, as it still requires careful range checking on the 0xF8-0xFF cases to prevent overflow bugs. Performance considerations sparked debate over SIMD compatibility, with kstenerud claiming similar approaches fail with vectorized instructions while jaen and nine_k contended that bijou64's length-prefix design is actually SIMD-friendly. The conversation also touched on whether varints are overrated for most use cases, with hansvm suggesting that zstd compression on flat binary formats often outperforms varint encoding for both speed and size.
