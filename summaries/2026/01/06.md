## [I/O is no longer the bottleneck? (2022)](https://stoppels.ch/2022/11/27/io-is-no-longer-the-bottleneck.html)
**Score:** 213 | **Comments:** 104 | **ID:** 46506994

> **Article:** The article "I/O is no longer the bottleneck?" (2022) argues that with the advent of high-speed NVMe SSDs, the performance bottleneck for many applications has shifted away from storage I/O. The author demonstrates that modern NVMe drives can saturate a single CPU core with data, making the core's processing speed or memory bandwidth the new limiting factor. The piece uses a practical example of parsing a large dataset to show that the CPU, not the disk, is what ultimately caps performance. It concludes that developers should now focus on optimizing CPU-bound tasks and memory access patterns rather than solely focusing on minimizing disk I/O.
>
> **Discussion:** The Hacker News discussion largely agrees with the article's premise but adds significant nuance and context. A central theme is that the classic "I/O bottleneck" was historically about slow mechanical hard drives with high seek times, which is a very different problem from the high-throughput sequential reads of modern SSDs. Commenters point out that the bottleneck has now shifted to other areas:

*   **CPU and Memory Bandwidth:** The primary new bottleneck is identified as the speed at which a single CPU core can ingest data (memcpy speed), often limited to around 6 GB/s on x86 architectures, which is slower than what top-end NVMe drives can provide. This makes memory access and data parsing (e.g., JSON) the new performance ceiling. This led to a discussion on "zero-copy" serialization formats that can bypass this limit.
*   **Cloud and Latency:** Several users countered that I/O is still a major bottleneck, especially in cloud environments where network-attached storage and virtualized filesystems can have poor performance. Others reframed the issue as one of latency versus throughput, arguing that most software is slow due to serial operations and waiting for I/O, rather than raw speed limits.
*   **Software Bloat:** A recurring sentiment is that despite hardware advancements, modern software (including operating systems and applications) often feels slower, which is attributed to poor design, inefficient code, and waiting for I/O on the main GUI thread.

Overall, the consensus is that while raw storage throughput is no longer the primary constraint for local applications, the bottleneck has simply moved to the CPU, memory, and network, with software design playing a critical role in perceived performance.
