## [I/O is no longer the bottleneck? (2022)](https://stoppels.ch/2022/11/27/io-is-no-longer-the-bottleneck.html)
**Score:** 253 | **Comments:** 123 | **ID:** 46506994

> **Article:** The article "I/O is no longer the bottleneck?" (2022) argues that with the advent of high-speed NVMe SSDs, the traditional performance bottleneck of disk I/O is shifting. The author demonstrates that on modern hardware, sequential read speeds from storage can exceed the rate at which a single CPU core can process that data. The core limit is identified as the CPU's memory bandwidth (the "memcpy" speed), which for a single x86 core is around 6 GB/s, while top-tier NVMe drives can deliver over 10 GB/s. This means that even perfectly optimized code can be limited by the CPU's ability to ingest data, not the disk's ability to provide it. The article suggests this shift requires a re-evaluation of software design assumptions that were built around the slow I/O of older hardware.
>
> **Discussion:** The Hacker News discussion largely validates the article's premise but adds significant nuance and context. A central theme is that the concept of an "I/O bottleneck" was never just about sequential reads. Commenters point out that for traditional spinning hard drives, the real bottleneck was random access and seek times, a problem that still exists for databases and other applications that don't read data linearly.

The conversation then deepens to identify the new primary bottlenecks. Several users highlight that the real limit is often the CPU's memory bandwidth per core, not just for raw data copying but also for parsing complex formats like JSON. This leads to a discussion on zero-copy serialization formats, which can bypass this CPU limit by allowing the processor to skip irrelevant data, leading to massive performance gains.

Other key points of discussion include:
*   **Cloud vs. Bare Metal:** A recurring point is that while hardware has advanced, cloud environments often reintroduce I/O bottlenecks through virtualization and throttling, making the article's findings more applicable to dedicated hardware.
*   **Latency vs. Throughput:** One user reframes the issue, arguing that most software is slow not because of hardware limits, but due to poor handling of latency, emphasizing the need for parallelization and batching.
*   **Software Performance Paradox:** A few commenters touch on the irony that despite faster hardware, modern software (like some desktop apps) often feels slower, attributing this to inefficient programming practices like blocking I/O on GUI threads.
*   **Future Architecture:** The discussion also speculates on future systems where the line between memory and storage blurs, with persistent memory creating a "CPU <-> Caches <-> Storage" architecture that could fundamentally change software design.
