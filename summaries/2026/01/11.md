## [Finding and fixing Ghostty's largest memory leak](https://mitchellh.com/writing/ghostty-memory-leak-fix)
**Score:** 349 | **Comments:** 69 | **ID:** 46568794

> **Article:** The article by Ghostty creator Mitchell Hashimoto details the process of finding and fixing a significant memory leak in the terminal emulator. The leak was located in the `PageList`, a data structure managing terminal scrollback memory. The root cause was a subtle bug in how "non-standard" memory pages (allocated with sizes different from the common default) were handled during a memory pool pruning operation. When the oldest page was pruned to make room for a new one, the code failed to correctly account for the non-standard size, leading to a mismatch between the memory being freed and the memory pool's internal tracking. This resulted in the memory never being deallocated. The fix was simple: if the page to be pruned is non-standard, it is simply discarded rather than being recycled back into the pool, ensuring the memory management remains consistent.
>
> **Discussion:** The Hacker News discussion was overwhelmingly positive, with users praising the author for the detailed write-up and the successful fix. A key theme was the critical importance of a "reliable reproduction" case from a user, which was instrumental in allowing the maintainer to finally locate and resolve the bug. This was contrasted with a previous comment from the maintainer expressing frustration over the lack of a reproducible case, which some users saw as a contradiction but was clarified as a shift from being unable to find the bug to being able to fix it once a repro was provided.

Several users engaged in technical deep-dives. One commenter offered alternative solutions to the fix, such as tagging memory allocations, which led to a discussion about the trade-offs between code simplicity and architectural robustness. The author defended the chosen fix as the simplest and most effective path forward, consistent with the existing design. Another user questioned the choice of data structure, suggesting a circular buffer, but the author noted this was an initial approach that was changed to avoid large memory allocations or copies. The discussion also touched on the use of AI for generating visualizations in the blog post and a tangential user-reported bug related to drag-and-drop in tmux.
