## [Finding and fixing Ghostty's largest memory leak](https://mitchellh.com/writing/ghostty-memory-leak-fix)
**Score:** 505 | **Comments:** 109 | **ID:** 46568794

> **Article:** The article by Mitchell Hashimoto details the process of finding and fixing a significant memory leak in the Ghostty terminal emulator. The leak was located in the `PageList`, a data structure responsible for managing terminal screen content (scrollback). The bug occurred when the terminal handled "non-standard" page sizes, which can be caused by certain UTF-8 characters or specific terminal operations.

The root cause was a flaw in the memory pool's logic for reusing pages. When a non-standard page was pruned from the pool and a new page of a different size was needed, the logic incorrectly attempted to reuse the pruned page, leading to memory being allocated with the wrong size and causing a leak. The fix was simple: ensure that a non-standard page is never reused if a new page of a different size is required, forcing it to be properly freed and a new one allocated instead. The article also discusses the use of AI to generate visualizations for the post.
>
> **Discussion:** The Hacker News discussion was overwhelmingly positive, praising the article's accessibility and the author's transparency. Key themes included:

*   **Appreciation for the Write-up:** Users lauded the post for being a clear and insightful look into a complex systems-level bug, even for those unfamiliar with terminal emulators.
*   **Clarification on Bug Severity and Attitude:** Some commenters perceived a contrast between the author's previous comments on the bug report and the tone of the article. The author (mitchellh) clarified that his position was consistent: he was sure the leak existed but was blocked by the lack of a reliable reproduction. The bug was rare and limited in impact ("largest" referred to leak size, not user impact), and the situation only changed once a user provided a solid repro case.
*   **Technical Alternatives and Questions:** Several users proposed alternative solutions to the fix, such as using a circular buffer, tagging memory blocks, or checking memory ownership. The author and other commenters explained that the chosen fix was the simplest and most effective, and that other approaches would add unnecessary complexity or performance overhead without sufficient data to justify them.
*   **User Experience and Impact:** Users confirmed the bug's real-world impact, noting that it affected them even without using the "Claude Code" tool that was suspected to trigger it more frequently. Others expressed relief and shared that they had experienced similar OOM crashes that they hoped were now resolved.

*   **Historical Context:** A user mused about what the web development landscape would look like if Haxe had gained popularity over TypeScript, suggesting it might have led to faster, native tooling much earlier. This was countered by the argument that TypeScript's success was due to its providing a practical migration path from JavaScript, rather than creating a new, separate ecosystem.

## [Show HN: Librario, a book metadata API that aggregates G Books, ISBNDB, and more](https://news.ycombinator.com/item?id=46571149)
**Score:** 115 | **Comments:** 39 | **ID:** 46571149

> **Project:** Librario is a new book metadata API that acts as an aggregator, pulling data from multiple sources like Google Books and ISBNDB. The project's goal is to provide a unified and more comprehensive source of book information by combining these different providers. It is currently in a pre-1.0 state with a demo server that may be unstable, but the author has built a Go SDK and is considering a TypeScript SDK. There are also mockups for a future user interface.
>
> **Discussion:** The HN community's response was largely positive and constructive, with users appreciating the effort while pointing out the significant challenges in this domain. A recurring theme was the difficulty of sourcing comprehensive data, with one user noting that Amazon is a major source but is tedious to scrape. Several suggestions were made for additional data sources to improve the API's coverage, including BookBrainz, Library of Congress, and especially Wikidata, which was highlighted as a powerful tool for aggregating identifiers from numerous libraries and catalogues.

Users also raised important technical and use-case considerations. Key questions included how the API handles books without ISBNs (currently unsupported, but planned), supports languages other than English (works for at least Portuguese), and manages data provenance to prevent overwriting manually corrected data (a future plan). A specific, unmet user need was identified: a service for tracking upcoming book releases from favorite authors. The discussion also touched upon the incompleteness of existing databases, as demonstrated by a user's failed search for a specific book on BookBrainz.
