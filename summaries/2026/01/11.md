## [Finding and fixing Ghostty's largest memory leak](https://mitchellh.com/writing/ghostty-memory-leak-fix)
**Score:** 382 | **Comments:** 78 | **ID:** 46568794

> **Article:** The article by Ghostty terminal developer Mitchellh details the process of finding and fixing a significant memory leak. The leak occurred within the `PageList`, a data structure managing terminal scrollback memory. The root cause was a mismatch between "standard" and "non-standard" page sizes. When a non-standard page (e.g., one created by a specific application like Claude Code) was added, the memory pool's logic for pruning and reusing pages would fail. It would incorrectly try to resize a non-standard page to a standard size, leading to memory being allocated but never freed.

The fix was simple: explicitly check if the page to be reused is non-standard. If it is, the code now correctly prunes (deletes) it instead of attempting to reuse it. This prevents the memory leak while maintaining the existing, performance-oriented design of the memory pool.
>
> **Discussion:** The Hacker News discussion was highly positive, with users praising the clear, accessible write-up and the successful debugging effort. A key theme was the critical value of a reliable reproduction case, which was ultimately provided by a user and allowed the developer to pinpoint the issue.

Some minor technical debates emerged:
*   **Fix Simplicity vs. Alternatives:** One user proposed alternative, more complex fixes (e.g., tagging memory, immutable sizes). Another user countered that the chosen fix was superior because it was simple, non-invasive, and solved the immediate problem without adding unnecessary complexity.
*   **Performance Trade-offs:** A user questioned why the fix didn't attempt to reuse a non-standard page if a new non-standard page was needed. The developer responded that the current logic is well-supported by benchmarks and that changing it would be a larger architectural shift, so the simple, correct fix was preferred for now.
*   **Developer Communication:** A user perceived a contrast between the developer's previous comments (which noted the difficulty in reproducing the bug) and the final blog post. Other users clarified that there was no contradiction; the developer had always believed a leak existed but was blocked by the lack of a reliable reproducer, which was the key to unlocking the fix.
