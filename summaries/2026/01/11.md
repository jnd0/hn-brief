## [Finding and fixing Ghostty's largest memory leak](https://mitchellh.com/writing/ghostty-memory-leak-fix)
**Score:** 434 | **Comments:** 94 | **ID:** 46568794

> **Article:** This article by Ghostty creator Mitchell Hashimoto details the process of finding and fixing a significant memory leak in the terminal emulator. The leak occurred in the `PageList`, a data structure managing terminal screen content (scrollback). The root cause was a mismatch between "standard" and "non-standard" memory page sizes. When a non-standard page was created (e.g., due to specific UTF-8 characters or features like those used by Claude Code) and the scrollback buffer needed to be pruned, the system would miscalculate the memory to free, leading to a leak. The fix was simple: if the oldest page to be pruned is non-standard, it is simply removed instead of being recycled, avoiding the faulty size calculation.
>
> **Discussion:** The Hacker News discussion was overwhelmingly positive, praising the article's accessibility and the transparency of the debugging process. Key themes included:

*   **Appreciation for the Write-up:** Users lauded the post for being a clear and insightful look into a complex systems-level problem, even for those unfamiliar with terminal emulators.
*   **Clarification on Bug Severity and Attitude:** Some users perceived a contrast between the blog post and a previous comment from the maintainer. Hashimoto clarified that he never denied the bug's existence; the issue was the lack of a reliable reproduction, which was only provided recently, allowing for the fix. He also noted the bug, while causing large memory leaks, had a limited user impact.
*   **Alternative Solutions and Simplicity:** Several commenters proposed alternative fixes (e.g., tagging memory, using a circular buffer). The prevailing sentiment, echoed by Hashimoto, was that the chosen fix was the simplest and safest. It involved minimal code changes and avoided altering core, well-performing logic without strong empirical data to justify it.
*   **Community Impact:** Users confirmed the fix was timely, with one reporting a similar OOM crash that seemed related. The discussion also highlighted the value of reliable bug reproductions from the community in resolving such issues.
