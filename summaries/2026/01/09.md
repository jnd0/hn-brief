## [What happened to WebAssembly](https://emnudge.dev/blog/what-happened-to-webassembly/)
**Score:** 308 | **Comments:** 285 | **ID:** 46551044

> **Article:** The article "What Happened to WebAssembly" addresses the perception that WASM has failed to live up to its initial hype. It argues that the "hype" itself was based on a misunderstanding of the technology's goals. The author clarifies that WebAssembly was never intended to replace JavaScript or build entire websites from scratch. Instead, its primary purpose is to act as a high-performance compilation target for languages like C++ and Rust, enabling computationally intensive tasks (e.g., video editing, 3D rendering, scientific simulations) to run in the browser at near-native speeds. The article concludes that WASM is succeeding in its intended niche but is not a general-purpose replacement for the JavaScript ecosystem.
>
> **Discussion:** The Hacker News discussion reveals a wide gap between WASM's practical achievements and the community's grander ambitions. A central theme is the disappointment that WASM hasn't fulfilled its "holy grail" potential as a universal cross-platform compile target that could replace everything, from desktop apps to server containers. However, several commenters counter that this was never the project's stated goal and that WASM is succeeding in its more modest role, such as accelerating specific performance-critical parts of applications like Figma or being integrated into the JS toolchain.

Key technical and practical hurdles were highlighted:
*   **Ecosystem Maturity:** The JavaScript ecosystem (React, TypeScript, Vite) is now so productive and performant that it remains the "no-brainer" choice for most web projects, diminishing the need for WASM's speed in many cases.
*   **Developer Experience:** There is a notable lack of documentation and accessible frameworks, making WASM a "power-user" tool rather than something for the average developer.
*   **Binary Size & Data Transfer:** WASM binaries can be large, leading to slow download times, and the lack of a standardized, efficient way to communicate with browser APIs (the host environment) creates performance bottlenecks and complexity.
*   **Governance:** Some commenters expressed concern over infighting and political disagreements within the standards bodies (e.g., WASI vs. AssemblyScript), which could hinder progress.

Despite the criticisms, many also pointed out WASM's quiet but significant success in areas like running legacy applications (Java, x86) in the browser and powering parts of the modern JS development toolchain itself.

## [Embassy: Modern embedded framework, using Rust and async](https://github.com/embassy-rs/embassy)
**Score:** 298 | **Comments:** 133 | **ID:** 46547740

> **Article:** Embassy is a modern, asynchronous framework for embedded systems development using Rust. It provides a unified, async-first approach to building firmware for various microcontrollers (like STM32, nRF52, and ESP32), abstracting away hardware differences and offering features like networking and BLE stacks. The framework is designed to work without a heap and provides a low-cost concurrency model suitable for single-core MCUs, effectively replacing the need for a traditional RTOS in many scenarios.
>
> **Discussion:** The HN discussion is largely positive, with many users praising Embassy for making embedded Rust development smoother and more powerful. Key themes include:

*   **Power of Async Rust:** Commenters highlight that Embassy's async model is a major advantage, enabling efficient concurrency on single-core chips without the complexity of a full RTOS or multi-threading. It's seen as a significant step up from the blocking, single-threaded mindset of hobbyist platforms like Arduino.
*   **Ecosystem Friction and Philosophy:** A notable counterpoint is raised about the ecosystem's heavy shift towards Embassy. This creates friction for developers who prefer not to use async, as it can feel like the community is splitting along an async vs. "blocking" line. Some worry this is not ideal for the still-nascent Rust embedded scene.
*   **Maturity and Adoption:** While some users note that certain components (like the Rust BLE stack) are still in early development with changing APIs, others point to its serious adoption, mentioning Microsoft is using it for an Embedded Controller project.
*   **Comparison to Alternatives:** RTIC (Real-Time Interrupt-driven Concurrency) is mentioned as another powerful option that uses the interrupt controller as a scheduler. Embassy's own `InterruptExecutor` serves a similar purpose, blurring the lines between the two approaches.
*   **Practical Use and Learning:** Users share positive experiences using Embassy for projects like LoRa relays and guitar amp controllers. A YouTube channel is shared as a great learning resource for getting started.
