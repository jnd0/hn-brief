## [Go.sum is not a lockfile](https://words.filippo.io/gosum/)
**Score:** 163 | **Comments:** 73 | **ID:** 46537095

> **Article:** The article "Go.sum is not a lockfile" by Filippo Valsorda argues that `go.sum` does not function as a true lockfile (like `package-lock.json` or `Cargo.lock`) and that developers should not treat it as one. The author explains that `go.sum` is a critical security mechanism for ensuring the integrity of dependencies by storing cryptographic checksums. Its primary purpose is to detect and prevent changes to upstream modules (e.g., if a version tag is moved or a module is compromised), causing builds to fail if the fetched content doesn't match the stored hash.

In contrast, the article posits that `go.mod` is the file that defines the dependency graph and versions. Due to Go's module system and minimum version selection, `go.mod` already provides reproducible builds by locking in the exact versions used. Therefore, `go.sum` is not needed for reproducibility, but is essential for security. The author concludes that `go.sum` should be committed, but its role is distinct from that of a traditional lockfile.
>
> **Discussion:** The discussion on Hacker News reveals a nuanced and sometimes contentious debate about Go's dependency management, centering on the roles of `go.mod` and `go.sum`.

A primary point of clarification is the function of `go.mod`. Several commenters correct the notion that `go.mod` only specifies minimum versions. They explain that Go's Minimum Version Selection (MVS) algorithm, combined with the default `-mod=readonly` flag, ensures that the exact versions listed (directly or indirectly) are used for a build. This makes `go.mod` functionally equivalent to a lockfile for reproducibility, a key distinction from other ecosystems where version ranges are resolved to the latest available.

The security role of `go.sum` is widely affirmed. Commenters agree with the article's main thesis: `go.sum` is a security layer, not a reproducibility tool. It prevents "supply chain" attacks by ensuring that the code built is the exact code intended, guarding against malicious or accidental changes to upstream repositories.

However, practical frustrations and edge cases were also highlighted:
*   **Tooling Lag:** A significant complaint is that official CI/CD tools, like GitHub's `actions/setup-go`, still rely on `go.sum` for caching, which is seen as incorrect and outdated. This points to a disconnect between the Go tooling's evolution and the practices of major platforms.
*   **Dependency Hell:** A recurring theme is the pain caused by large libraries (e.g., AWS, Kubernetes) that create massive, unstable dependency graphs. Even unused sub-packages can force version upgrades, leading to build failures. This makes some developers feel that Go's model is naive and that vendoring or splitting dependencies into separate binaries are more robust solutions.
*   **Private Registries:** An edge case was raised where a private proxy might serve a modified module without changing its version tag. In this scenario, `go.mod` wouldn't change, but the build would be compromised. This highlights a scenario where the security guarantees of `go.sum` are critical, but also where CI caching based only on `go.mod` could be dangerous.

In essence, while the community largely agrees on the theoretical distinction between `go.mod` (reproducibility) and `go.sum` (security), real-world tooling, ecosystem issues, and edge cases create significant practical challenges and frustrations.
