## [What happened to WebAssembly](https://emnudge.dev/blog/what-happened-to-webassembly/)
**Score:** 283 | **Comments:** 259 | **ID:** 46551044

> **Article:** The article "What happened to WebAssembly" addresses the perception that WASM has failed to live up to its initial hype. The author argues that the "disappointment" stems from a misunderstanding of its purpose. The original goal of WASM was not to replace JavaScript for building entire websites, but to provide a high-performance compilation target for computationally intensive tasks (like Figma's design tools or games) that run alongside JavaScript. The article highlights several remaining challenges that have slowed adoption: the difficulty of direct DOM manipulation from WASM (requiring a JS bridge), the lack of a standardized garbage collector for garbage-collected languages (though WasmGC is now changing this), and the large binary sizes compared to JavaScript. Ultimately, the piece concludes that WASM is succeeding in its niche as a performance tool but is not the "JavaScript killer" some expected.
>
> **Discussion:** The Hacker News discussion largely mirrors the article's themes but offers a more nuanced view of WASM's current state. A central point of debate is the definition of WASM's success. Several commenters express disappointment that WASM hasn't replaced JavaScript for general web development, viewing this as the technology's primary promise. However, others counter that this was never the official goal and that WASM is quietly succeeding as a backend for performance-critical libraries and build tools within the existing JavaScript ecosystem.

The conversation highlights key technical hurdles that have limited adoption:
*   **Ecosystem Maturity:** Commenters point to poor documentation, a steep learning curve (especially with Rust), and a lack of mature, high-level frameworks for building user interfaces directly in WASM.
*   **Performance Bottlenecks:** The overhead of data marshalling between WASM and JavaScript is a major issue, making it inefficient for fine-grained interactions. Furthermore, large binary sizes are a significant problem for users with slow internet connections, as WASM cannot stream-execution like JavaScript.
*   **Developer Experience:** The JS ecosystem (React, TypeScript, Vite) is described as "so incredibly good" and productive that the performance gains from WASM aren't worth the added complexity for most applications.

There is also a split in vision for WASM's future. While some see its ultimate potential as a universal cross-platform compile target for servers and desktops (e.g., replacing containers with WASI), others are more pragmatic, viewing it as a powerful tool for specific use cases like image editing, running legacy applications (Java, x86 binaries) in the browser, or accelerating specific parts of a web app.

Commenters also discussed the user's role in the vulnerability. The exploit required the user to have "always allow" enabled for commands, a high-risk configuration. The consensus was that true security for running untrusted code requires more robust measures, like ephemeral virtual machines, rather than simple container-based sandboxes which can be theater.
