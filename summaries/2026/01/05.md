## [Maybe comments should explain 'what' (2017)](https://www.hillelwayne.com/post/what-comments/)
**Score:** 187 | **Comments:** 180 | **ID:** 46486780

> **Article:** The 2017 article by Hillel Wayne argues against the common practice of using comments to explain the "what" (i.e., what the code is doing). He contends that this information should be conveyed through the code's structure itself—using better variable names, clearer function calls, and more expressive abstractions. The goal is to make the code self-documenting. Comments, in his view, are better reserved for explaining the "why" behind a piece of code: the business logic, historical reasons, or non-obvious constraints that cannot be expressed through syntax alone. He believes this separation makes code more maintainable, as comments explaining "what" tend to rot and become misleading when the code changes, whereas the "why" often remains relevant.
>
> **Discussion:** The Hacker News discussion largely agrees with the article's premise but explores significant nuances and practical realities. A central theme is a strong backlash against "Uncle Bob" style programming, with many developers sharing frustrations about over-extraction of methods, which they argue harms readability by forcing them to navigate excessive indirection. The main points of debate are:

*   **The "What" vs. "Why" Distinction:** While many agree that "why" comments are most valuable, others argue the line is blurry. "What" comments are seen as essential for explaining complex algorithms, performance optimizations, or domain-specific business rules that are not obvious from the code itself.
*   **Code as Documentation:** A recurring point is that the ideal of self-documenting code has limits. Overly verbose names or excessive abstraction can be just as confusing as poorly commented code. The best approach is a balance, using clear code where possible and comments for what the code cannot express.
*   **Pragmatism over Dogma:** The most popular sentiment is that comments should be used pragmatically to aid understanding. The purpose of a comment is simply to explain the programmer's intent, regardless of whether it falls into a strict "what" or "why" category. Dogmatic rules against either type of comment are seen as unhelpful.

In essence, the community endorses the article's spirit but rejects a rigid interpretation, advocating for a balanced approach where code clarity is paramount but comments are a vital tool for providing context that code alone cannot.

## [Understanding the bin, sbin, usr/bin, usr/sbin split (2010)](https://lists.busybox.net/pipermail/busybox/2010-December/074114.html)
**Score:** 170 | **Comments:** 125 | **ID:** 46487921

> **Article:** The linked article is a 2010 email from the BusyBox mailing list that explains the historical reasons for the split between `/bin`/`/sbin` and `/usr/bin`/`/usr/sbin`. The author, Rob Landley, traces the lineage back to early Unix. The core reason was hardware limitations: `/bin` and `/sbin` contained essential tools needed to mount `/usr` and bring the system to a usable state. `/usr` would often be on a separate, network-mounted, or larger disk, and contained the rest of the operating system and user applications. The "s" in `sbin` denotes "system" binaries, intended for the administrator, not regular users. The article also clarifies the purpose of `/usr/local` as a place for custom-compiled software that wouldn't interfere with the distribution's package manager in `/usr`.
>
> **Discussion:** The Hacker News discussion is a mix of historical appreciation, frustration with legacy complexity, and modern context. A central theme is that many current Linux structures are historical artifacts whose original justifications (like small, expensive disks) are no longer relevant, leading to comments like "because it's always been that way." Users express amusement and nostalgia, noting that modern immutable operating systems have "come full circle" back to the read-only root filesystem concept of early floppy-based Unix systems.

There is a significant debate about whether this legacy is necessary. While some defend the structure as logical for its time (separating essential boot files from the rest of the system in a multi-user environment), others argue it's needless complexity maintained by standards bodies. The discussion also touches on modern attempts to simplify this, such as Arch Linux's move to merge binaries into `/usr` and alternative filesystem layouts from projects like GoboLinux, NixOS, and Guix. The conversation frequently broadens to a meta-commentary on how technical standards and cruft accumulate over time, comparing it to the history of the User-Agent string or the proliferation of constructed languages like Esperanto.

## [Why does a least squares fit appear to have a bias when applied to simple data?](https://stats.stackexchange.com/questions/674129/why-does-a-linear-least-squares-fit-appear-to-have-a-bias-when-applied-to-simple)
**Score:** 127 | **Comments:** 29 | **ID:** 46491821

> **Article:** The article (a Stack Exchange question) addresses why a standard linear least squares regression can appear biased or "wrong" when visually inspected on simple 2D data. The author observes that when you plot a regression line of y on x, it is different from the line you get by regressing x on y, and neither looks like the "true" line that visually bisects the data points. The core issue is that Ordinary Least Squares (OLS) minimizes the sum of the *vertical* distances (squared errors in y) between the data points and the line, assuming x is error-free. This mathematical constraint creates an asymmetry that often results in a line that does not align with the visual center of the data cloud, especially when the data has a high correlation or the axes are scaled similarly.
>
> **Discussion:** The discussion centers on explaining the mathematical and conceptual reasons for this visual bias, and exploring alternative methods. A primary theme is the distinction between different regression models based on their underlying assumptions about error. Commenters note that OLS assumes noise only in the dependent variable (y), while our visual intuition often assumes noise in both variables (x and y). This "Total Least Squares" approach, which minimizes the perpendicular distance to the line, can be achieved using methods like Principal Component Analysis (PCA) or Deming regression.

Key points raised include:
*   **Different Loss Functions:** The fundamental difference is what is being minimized. OLS minimizes vertical errors, while PCA minimizes the shortest (perpendicular) distance from points to the line.
*   **Practical Solutions:** Deming regression is frequently suggested as a solution that accounts for errors in both x and y, requiring an estimate of the relative variance of the errors. PCA is also mentioned as a way to find the line that best fits the data's overall variance.
*   **Real-World Context:** Many commenters argue that the OLS assumption (no error in x) is often unrealistic. However, others counter that in many scenarios, such as sensor data with a stable clock, the noise in the measurement (y) is orders of magnitude larger than the "noise" in the time coordinate (x), making OLS perfectly adequate.
*   **Interpretation and Nuance:** It's clarified that while the regression line may look visually biased, it is statistically "unbiased" for predicting y given a specific x. The "bias" is a feature, not a bug, of the model's specific goal. The discussion also touches on how to visually verify a fit by checking that residuals are symmetrically distributed around the line.
