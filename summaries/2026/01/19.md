It explains how user data (posts, profiles, likes) is stored in a Personal Data Server (PDS), which functions like a file system where data is organized into repositories and records. The author argues that this architecture decouples the social graph and data storage from the application layer, allowing users to own their data while still participating in a public, aggregated social network. The article uses the analogy of a web server serving static files to explain how a PDS works and highlights tools like `pdsls.dev` that allow users to browse their PDS data directly.
>
> **Discussion:** The discussion on Hacker News centered on the feasibility, philosophy, and technical architecture of the AT Protocol. Reactions were mixed, ranging from enthusiastic support to deep skepticism.

A significant portion of the debate focused on the practicality of user-owned data. One commenter criticized the protocol as a "faux-libertarian" scheme that still relies on centralized entities (like Bluesky) for aggregation and monetization, arguing that true change requires simple, consumer-friendly home servers. However, others countered that Bluesky's Personal Data Servers (PDS) are already live and easy to set up, with tools like PDSmoover facilitating data migration. Author Dan Abramov clarified that hosting user data is actually cheap, and the real costs for platforms lie in processing media and aggregating feeds, regardless of where the data is stored.

Technically, the discussion compared AT Protocol to other decentralized systems. Some commenters felt the protocol was over-engineered, suggesting a simpler, hash-based lookup system similar to SSB or ANProto would be better. The author defended the design, noting that it prioritizes solving the problem of aggregating public data across many users—a challenge he feels projects like Solid or remoteStorage have not adequately addressed. Comparisons were drawn to the Solid project, with Abramov noting that Solid focuses on private data and RDF, while AT focuses on public data and schema-defined records (Lexicons).

Finally, the conversation touched on broader themes of data permanence and privacy. Commenters noted that because PDS data is public and replicated, users must practice good operational security, as deleted data may persist elsewhere. While some viewed the protocol as part of a hopeful trend toward "local-first" and humane computing, others remained concerned about the technical barriers to entry for the average user.
