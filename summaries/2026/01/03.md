## [The rsync algorithm (1996) [pdf]](https://www.andrew.cmu.edu/course/15-749/READINGS/required/cas/tridgell96.pdf)
**Score:** 171 | **Comments:** 26 | **ID:** 46466734

> **Article:** The linked document is Andrew Tridgell's 1996 paper detailing the "rsync algorithm." It presents an efficient method for synchronizing files between two computers (a sender and a receiver) over a network. The core innovation is avoiding the transfer of entire files when only small parts have changed. The algorithm works by having the receiver compute rolling checksums for chunks of its existing file and sending these checksums to the sender. The sender then compares these against checksums of the new file to identify which blocks are identical and which are new, transferring only the new or changed blocks. The paper includes benchmarks demonstrating significant bandwidth savings compared to transferring the whole file, especially for small modifications in large files.
>
> **Discussion:** The discussion praises the rsync algorithm as a classic example of elegant, problem-solving computer science. Many commenters share personal success stories of using rsync for tasks like remote backups, NAS setup, and efficiently updating large virtual machine images, with one user noting it saved them over a terabyte of data transfer.

A key technical point raised is that rsync's default behavior relies on file size and modification time to detect changes. This can fail for reproducible builds (like those from NixOS) where timestamps are fixed, requiring the use of the `--checksum` flag to ensure correctness.

The conversation also highlights the broader impact of rsync's author, Andrew Tridgell, noting his reverse-engineering of the SMB protocol for Samba and his role in the creation of git. There is a minor debate on the legal risks of reverse-engineering protocols versus software. Finally, users share their favorite command-line flags and mention alternatives like Google's CDC file transfer.
