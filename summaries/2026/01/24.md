## [Microsoft mishandling example.com](https://tinyapps.org/blog/microsoft-mishandling-example-com.html)
**Score:** 232 | **Comments:** 81 | **ID:** 46731996

> **Article:** The article details a significant security flaw in Microsoft's Autodiscover protocol used by Outlook. The system is designed to automatically configure email clients by discovering server settings for a given domain. However, it has a "fallback" mechanism that is dangerously broad: if the primary method fails (e.g., for `example.com`), the protocol attempts to find configuration at broader, less specific domains like `autodiscover.example.com`, and ultimately `autodiscover.com`.

The author demonstrates that Microsoft's own systems have effectively claimed the reserved IANA domain `example.com`. As a result, any organization using Outlook that does not have a properly configured `autodiscover` subdomain may have their login prompts, credentials, and configuration requests routed to Microsoft's servers, which then incorrectly route them to the mail servers of Sumitomo Electric Industries (`sei.co.jp`). This creates a significant data leak, where test credentials and other sensitive authentication data are sent to a third-party corporation.
>
> **Discussion:** The Hacker News discussion largely validates the article's findings and expands on the broader context of Microsoft's architectural decisions. The consensus is that the Autodiscover protocol is fundamentally flawed, described by one user as "braindead" for its aggressive and insecure fallback behavior.

Key themes in the discussion include:

*   **Protocol Flaws:** Users point to an Akamai blog post that details exactly how the protocol works. The core issue is that if a user enters an email like `user@example.com`, Outlook attempts to connect to `autodiscover.example.com`. When that fails, it falls back to `autodiscover.com`, which is a registered domain controlled by Microsoft. This design prioritizes convenience over security, effectively leaking credentials to Microsoft's servers.
*   **Privacy and Credential Handling:** Several commenters expressed alarm that Outlook sends full credentials (username and password) to Microsoft's servers during the auto-discovery process, even for third-party email providers. While some argue this is standard for SaaS products, others view it as a breach of privacy expectations, as users may not realize their passwords are being sent to Microsoft before they even connect to their own mail server.
*   **Historical Incompetence:** The discussion frames this incident as part of a pattern of poor engineering decisions by Microsoft. Users cited Microsoft's past recommendation to use `.local` for Active Directory, which later conflicted with Multicast DNS (mDNS) standards, causing issues for Linux and VMware environments. This history has led to a lack of trust in Microsoft's domain management and protocol design.
*   **IANA Reserved Domains:** There was a debate about the use of reserved domains like `.example`. While the article warns against using them, some users argued that `.example` is safer than `example.com` because it lacks the specific fallback vulnerability to `autodiscover.com`. However, the general advice was to use made-up, impossible domains to avoid any risk of DNS misconfiguration leaking data to a real server.
