## [Many Small Queries Are Efficient in SQLite](https://www.sqlite.org/np1queryprob.html)
**Score:** 112 | **Comments:** 81 | **ID:** 46742635

> **Article:** The linked article from SQLite.org argues that executing many small, separate queries can be more efficient than a single complex query, particularly in an embedded database context. It uses the timeline feature of the Fossil SCM (which uses SQLite) as a case study. The author, D. Richard Hipp, explains that while a single complex query might seem theoretically superior, the overhead of parsing, planning, and transferring large result sets can be significant. In contrast, many small, simple queries are often already optimized, cached, and executed with minimal overhead. The article demonstrates that for the Fossil timeline, which executes hundreds of queries per page load, the total time is dominated by application-level logic (like string formatting) rather than the database engine itself, making the "many queries" approach highly performant and simpler to manage.
>
> **Discussion:** The Hacker News discussion largely validates the article's premise, focusing on the reasons why SQLite is exceptionally fast for local, embedded operations. The key theme is the elimination of network overhead. Commenters explain that even a connection to a database on the same machine (localhost) involves significant latency from the network stack, kernel transitions, and data serialization. In contrast, SQLite is an in-process library, accessed via direct function calls within the application's memory space, which is orders of magnitude faster for small, frequent operations.

The conversation then pivots to the practical trade-offs and ideal use cases for SQLite. It's described as an "application file-format with a SQL interface," perfect for single-user applications, local services, or scenarios with low write concurrency. Several users shared positive experiences using SQLite in production for web applications and even in browser-based environments (via WASM), noting that it often outperforms native JavaScript objects at scale.

However, the discussion also highlights critical limitations.
