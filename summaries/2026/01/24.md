## [Many Small Queries Are Efficient in SQLite](https://www.sqlite.org/np1queryprob.html)
**Score:** 141 | **Comments:** 95 | **ID:** 46742635

> **Article:** The article "Many Small Queries Are Efficient in SQLite" argues that for applications using an embedded database, executing numerous small, separate SQL queries can be more performant than crafting a single, complex query. The author (from the SQLite team) demonstrates this using the Fossil SCM's timeline feature as a case study. The core thesis is that the overhead of a network round-trip is the primary bottleneck in traditional client-server databases (like PostgreSQL or MySQL), making batched queries necessary. In contrast, SQLite, being an in-process library, has negligible function call overhead. Therefore, developers can prioritize code simplicity and readability by issuing simple, targeted queries for each piece of data needed, as the cost is often lower than the complexity of writing and optimizing a monolithic query.
>
> **Discussion:** The Hacker News discussion largely validates the article's premise but adds crucial context and clarifies the underlying mechanics. The consensus is that the performance advantage isn't about SQLite's query engine being magically faster, but about the elimination of network overhead.

A key theme is the distinction between network latency and local processing. Commenters explain that even a "localhost" connection to a database like MySQL involves significant overhead from the network stack, kernel transitions, and inter-process communication. SQLite, as an embedded library, bypasses this entirely with direct function calls within the same process memory space, making its "latency" orders of magnitude lower.

The conversation also explores the trade-offs and practical applications. While a single complex query can be more efficient for large-scale analytical work (as the query optimizer can find better execution plans), many application workloads consist of simple record retrievals where this optimization is less critical. SQLite is often described as an excellent "application file-format with a SQL interface," ideal for single-user or low-concurrency scenarios. Commenters shared experiences using SQLite in production for web apps and even in-browser with WebAssembly, finding it performs well until write concurrency becomes a major bottleneck, at which point migrating to a server like PostgreSQL is a straightforward upgrade path. The discussion also highlighted that this design philosophy is self-referential: SQLite's own version control system, Fossil, uses a SQLite database for storage.
