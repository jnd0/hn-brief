>
> **Discussion:** The Hacker News discussion largely validates the author's core frustration but offers a variety of solutions and alternative perspectives. The central theme is that while GitHub Actions can be painful, the pain can be mitigated with better practices and tools.

A primary point of agreement is the advice to keep logic out of workflow files. Many commenters echo the author's conclusion, stating that CI workflows should be simple "dumb" orchestrators that call external, locally-testable scripts. This practice prevents lock-in and makes debugging significantly easier.

The most prominent solutions discussed are:
1.  **Local Testing Tools:** The tool `act` is frequently mentioned as a way to simulate GitHub Actions locally, but several users, including the author, noted its limitations and that it doesn't perfectly replicate the CI environment for complex workflows.
2.  **Interactive Debugging:** A major point of discussion is the lack of an easy way to SSH into a failed runner. Commenters highly recommend tools like `action-tmate` (and its successor `upterm`) to get an interactive shell on a running CI job, which dramatically simplifies debugging environment-specific issues. Several people argue this "Rebuild with SSH" feature should be built into GitHub Actions natively.
3.  **Better Tooling Management:** Some commenters suggest that the root cause of such issues is often inconsistent tooling between local and CI environments. They recommend using tools like `mise` or `nix` to ensure that the exact same versions of dependencies are available everywhere, thus preventing the "works on my machine" problem.

Finally, there is a debate on whether this is a unique problem to GitHub Actions. While some argue other platforms like GitLab or SourceHut have better debugging features, others contend that the fundamental challenges of CI are universal. The consensus is that a well-designed CI pipeline, regardless of the platform, should be robust, testable, and easy to debug.
