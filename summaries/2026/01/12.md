*   **Instability:** The C++ API is not stable, making it difficult for frontends (like PHP or Rust) to upgrade their LLVM versions without significant effort. While the C API is better, it's not a panacea.
*   **Documentation and Complexity:** The internal architecture, especially components like SelectionDAG, is poorly documented. This forces developers to rely on reading the source code to understand the precise semantics of operations, leading to fragile implementations.
*   **Compilation Performance:** LLVM's compile times are a major pain point, especially for languages like Rust that rely heavily on it. The article notes that the initial promise of fast compilation has been eroded as the system has grown.
*   **Optimization Bugs:** The optimizer can be a source of subtle and hard-to-diagnose bugs, and the code review process within the LLVM project can be challenging for contributors.

The author concludes that while LLVM is an indispensable tool, its "bad parts" create significant ongoing maintenance burdens for anyone building a language on top of it.
>
> **Discussion:** The Hacker News discussion largely validates the article's points, with many commenters sharing their own experiences with LLVM's complexities. The conversation centers on a few key themes:

*   **Compilation Speed:** This is a major point of frustration. Several users note that LLVM, once touted as faster to compile than GCC, is now extremely slow. *   **Stability and Maintenance Burden:** Commenters, including one who maintains a project in LLVM, confirm that the C++ API is a moving target, requiring significant work to rebase projects between versions. *   **Internal Complexity and Lack of Documentation:** The difficulty of working with LLVM's internals is a recurring point. One user specifically echoes the article's call for a comprehensive test suite starting from LLVM IR to help backend developers understand the expected semantics of operations.

*   **Safety and Auditing:** A more philosophical point was raised about the security implications of LLVM being a foundational, unauditable piece of software for "safe" languages like Rust. *   **Specific Optimization Issues:** The discussion touched on the article's example of a register pressure issue, with experts debating whether the problem lies with the Loop Invariant Code Motion (LICM) pass or the register allocator's ability to "rematerialize" values.
