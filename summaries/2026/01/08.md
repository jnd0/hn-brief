## [“Stop Designing Languages. Write Libraries Instead” (2016)](https://lbstanza.org/purpose_of_programming_languages.html)
**Score:** 237 | **Comments:** 216 | **ID:** 46525640

> **Article:** The article "Stop Designing Languages. Write Libraries Instead" argues that developers should focus on creating powerful libraries within existing, mature general-purpose languages rather than inventing new Domain-Specific Languages (DSLs). The author posits that a sufficiently expressive general-purpose language can provide all the benefits of a DSL through its library system.

The core reasoning is that libraries are more practical and less disruptive. Adopting a new library is a low-friction process that integrates with a developer's existing tooling, while a new language requires a complete ecosystem switch, which is a significant barrier. Furthermore, the author argues that the features of a language directly enable or constrain its libraries; a more powerful language leads to more elegant and usable libraries. The article uses Ruby on Rails as a prime example of a "language-like library" that succeeded precisely because of Ruby's unique features (e.g., metaprogramming, first-class functions), which allowed it to abstract away complexity for non-expert users and create a highly expressive DSL for web development.
>
> **Discussion:** The discussion on Hacker News was varied, with many commenters engaging with the article's provocative title and its core thesis.

A central theme was the debate over the definition and utility of DSLs. One commenter suggested that libraries that construct Abstract Syntax Trees (ASTs) are functionally equivalent to DSLs, with the added benefit of better LLM support for popular languages like Python. Another argued that the line between a "language" and a "library" is blurry, citing a JSON parsing library as an example of a library that defines a language.

Several users debated the practicality of the article's premise. Some agreed that adopting a new library is far less risky than a new language, citing factors like hiring and tooling maturity. However, others pushed back, arguing that the Sapir-Whorf hypothesis holds true for programming—that the language itself fundamentally shapes how developers think about and solve problems, something a library cannot fully replicate. The discussion also touched on the role of standard libraries, with one user criticizing the inclusion of features like testing frameworks in runtimes for stifling innovation, while another defended it as a matter of convenience.

Finally, a significant portion of the discussion focused on the article's central example, Ruby on Rails. One commenter argued that Ruby's decline is undeniable and that its popularity was largely tied to Rails, not the language itself. This was countered by another user who asserted that Ruby and Rails remain massive and influential, powering major websites like Shopify and GitHub.
