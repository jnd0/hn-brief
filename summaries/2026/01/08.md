## [“Stop Designing Languages. Write Libraries Instead” (2016)](https://lbstanza.org/purpose_of_programming_languages.html)
**Score:** 244 | **Comments:** 238 | **ID:** 46525640

> **Article:** The 2016 article "Stop Designing Languages. Write Libraries Instead" argues that developers should focus on creating libraries for existing general-purpose languages rather than inventing new programming languages. The author contends that libraries are more practical because they leverage existing ecosystems, tooling, and developer knowledge, whereas new languages require users to abandon their entire environment. The piece uses Ruby on Rails as a prime example: it is not a new language but a library (framework) that provides a powerful Domain-Specific Language (DSL) for web development. This DSL is only possible because of Ruby's dynamic features (like `method_missing` and blocks), but the key takeaway is that the library is what delivers the value. The author asserts that powerful libraries can effectively create "languages" within a host language, offering the benefits of domain-specific expressiveness without the high adoption costs of a standalone language.
>
> **Discussion:** The Hacker News discussion is multifaceted, with users engaging with the article's premise, its title, and the broader philosophy of language and library design. A central theme is the blurry line between a "language" and a "library." Several commenters point out that powerful libraries often create internal Domain-Specific Languages (DSLs), effectively becoming languages themselves. The debate then splits on the utility of this approach: one side champions libraries for their convenience and lower barrier to entry, while others defend the creation of new, standalone languages, arguing that a well-designed language can fundamentally change how one thinks about a problem (the Sapir-Whorf hypothesis), something a library cannot fully achieve.

The discussion also heavily references the article's main example, Ruby on Rails. One commenter argues that Ruby's decline is proof that frameworks built on niche languages are risky, while another counters that the Ruby/Rails ecosystem remains robust and vital. Practical concerns are raised, such as the difficulty of hiring for new languages versus the ease of adopting a new library. The conversation also touches on the trend of languages integrating popular libraries into their standard libraries (e.g., Node.js's built-in test runner), with a split between those who see it as convenient and others who view it as stifling innovation. Finally, a meta-comment humorously predicts that many users would comment based on the title alone, a sentiment that seems to be validated by the diverse and sometimes contradictory points raised.
