## [Go.sum is not a lockfile](https://words.filippo.io/gosum/)
**Score:** 158 | **Comments:** 70 | **ID:** 46537095

> **Article:** The article "Go.sum is not a lockfile" by Filippo Valsorda argues that `go.sum` is not a lockfile, but rather a security and integrity mechanism. He explains that a lockfile's primary purpose is to ensure reproducible builds by pinning exact dependency versions for a project. In Go, this role is fulfilled by `go.mod` combined with the module system's "minimal version selection" (MVS). MVS deterministically selects the minimum required version of each dependency needed to satisfy all constraints, making builds reproducible without a separate lockfile.

`go.sum`, on the other other hand, serves as a security cache. It stores the cryptographic checksums of dependencies to detect accidental changes or malicious tampering (e.g., if a tag is moved to point to different code). Its purpose is to verify the integrity of the fetched code, not to lock its version. Therefore, `go.mod` is the true lockfile, and `go.sum` is a security manifest.
>
> **Discussion:** The discussion reveals a significant split between Go's design philosophy and developer expectations from other ecosystems. The core debate centers on whether `go.mod` is truly equivalent to a lockfile.

A major point of contention is the role of checksums. Many developers, particularly those from JavaScript or Python backgrounds, view cryptographic hashes as an essential part of a lockfile's function. They argue that without `go.sum`, there is no way to verify that the fetched dependencies are the exact, untampered versions intended. The article's premise that `go.sum` has "no effect on builds" is seen as dangerous by some, as it overlooks its critical security role in preventing supply chain attacks.

Another key theme is the difference in dependency resolution strategies. Commenters explain that Go's "minimal version selection" (MVS) is fundamentally different from the "latest compatible version" strategy used by tools like npm or Maven. This makes `go.mod` a reliable, deterministic lockfile by default, whereas other ecosystems require a separate lockfile to prevent upstream changes from breaking builds. However, some remain skeptical, pointing out that MVS can still upgrade transitive dependencies, which they feel undermines the "lock" aspect.

Finally, there is significant frustration with the practical implementation, especially in CI/CD. The `actions/setup-go` GitHub Action's default caching behavior is criticized for using `go.sum`'s modification time, which can lead to stale caches and broken builds. This practical friction leads some to argue that Go's reliance on "good citizen" dependencies is naive and that vendoring or more robust tooling is necessary to handle real-world complexities like private modules or amended tags.
