## [Why users cannot create Issues directly](https://github.com/ghostty-org/ghostty/issues/3558)
**Score:** 557 | **Comments:** 197 | **ID:** 46460319

> **Article:** The linked GitHub issue explains why the Ghostty terminal project does not allow users to create issues directly. Instead, all user reports (potential bugs, feature requests, and questions) must start in GitHub Discussions. The maintainers then manually triage these discussions; only once a report is fully understood, reproduced, and confirmed as an actionable bug or feature does a maintainer convert it into an Issue. The stated goal is to ensure the Issue tracker remains a "clean" list of work that is ready to be started, rather than a noisy mix of user errors, configuration problems, and unvetted ideas.
>
> **Discussion:** The Hacker News community had a mixed but analytical reaction to Ghostty's policy. The debate centered on whether this workflow is an efficient filter or simply shifting the triage burden without solving the underlying problem.

**The "Cleanliness" Argument**
Supporters argued that standard issue trackers often become overwhelmed by noise. They noted that the vast majority of user-submitted "bugs" are actually configuration errors or misunderstandings. By forcing these reports into Discussions first, maintainers can avoid a cluttered Issue list. Proponents believe this sets better expectations for users and allows for informal troubleshooting (e.g., solving memory leaks via conversation) without permanently clogging the bug tracker.

**The "Process Overhead" Argument**
Skeptics questioned the efficiency of the system. They argued that the time spent reading a report and deciding whether to convert it to an Issue is roughly the same as reading it and applying a "triage" label. Many suggested that simple tagging (e.g., "needs-investigation" vs. "ready-to-work") or using GitHub's "Projects" board could achieve the same separation of concerns without forcing users into a different UI. There was also concern that this creates friction for users, potentially causing them to abandon the software if they hit a bug (like a memory leak) that sits in Discussions rather than the Issue tracker.

**Platform Limitations**
Several users noted that GitHub Discussions and Issues are nearly identical in UI, making the distinction feel arbitrary. There was a call for GitHub to natively support this "funnel" workflow (e.g., allowing issues to be converted to discussions or vice versa more seamlessly) to formalize the process.
