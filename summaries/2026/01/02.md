## [Why users cannot create Issues directly](https://github.com/ghostty-org/ghostty/issues/3558)
**Score:** 675 | **Comments:** 244 | **ID:** 46460319

> **Article:** The article is a GitHub issue titled "Why users cannot create Issues directly" in the `ghostty-org/ghostty` repository. It explains the project's policy of not allowing users to create issues directly. Instead, all user reports, questions, and feature requests must start as GitHub Discussions. The maintainers then triage these discussions, and only when a "well-understood, actionable item" is identified do they convert it into an issue. The stated goal is to ensure that the issue tracker only contains work items that are ready for contributors, filtering out the "80-90%" of reports that are user errors, configuration problems, or misunderstandings. This process is based on the maintainers' years of experience managing open-source projects.
>
> **Discussion:** The Hacker News discussion centered on the merits and practicalities of Ghostty's workflow. The core debate was whether this two-step process is genuinely more efficient than using a single issue tracker with labels and tags.

Many commenters supported the philosophy, arguing that the issue tracker should be a clean, actionable backlog for contributors, not a noisy support forum. They felt that using Discussions first sets better cultural expectations for the type of conversation and keeps the issue threads focused. The distinction, some argued, is more than just technical; it's a process and communication choice that helps separate "support" from "work."

However, a significant counter-argument questioned the practical efficiency. Skeptics pointed out that the initial triage work of reading and understanding a report is the same regardless of whether it's a Discussion or an Issue. They argued that the same filtering could be achieved with labels (e.g., "needs-triage," "ready-to-work-on") and that the Ghostty method just adds an extra step of converting a discussion to an issue.

Other points raised included:
*   **Process vs. Tooling:** Some suggested using GitHub's "Projects" feature for the curated work board, while keeping issues for raw bug reports.
*   **Real-world Evidence:** One user cited the highly successful `curl` project as a counter-example that manages a high volume of issues effectively.
*   **A Tangible Problem:** A specific example of a memory leak investigation was raised, with the original reporter noting it was spread across multiple discussions and social media but had not yet been promoted to an official issue, highlighting a potential downside of the strict process.
