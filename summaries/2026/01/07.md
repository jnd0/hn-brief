## [High-Performance DBMSs with io_uring: When and How to use it](https://arxiv.org/abs/2512.04859)
**Score:** 125 | **Comments:** 25 | **ID:** 46517319

> **Article:** The paper "High-Performance DBMSs with io_uring: When and How to use it" analyzes the performance of the Linux `io_uring` asynchronous I/O framework for database management systems. It provides practical guidelines for its effective use, arguing that it is not a simple drop-in replacement for traditional I/O methods but requires a fundamental architectural shift. The key recommendations are: 1) Use a "ring-per-thread" model that overlaps computation and I/O within the same thread, rather than using a shared ring or a traditional I/O-worker model. 2) Only adopt `io_uring` after confirming I/O is the actual bottleneck. 3) Progressively use advanced features like batching, fixed buffers, zero-copy, and polling only after the basic async/batching architecture is in place. The paper demonstrates significant performance gains (e.g., increasing throughput to 300k tx/s) when these principles are applied, especially with features like NVMe passthrough (`OP_URING_CMD`).
>
> **Discussion:** The Hacker News discussion was largely positive, with users praising the paper for its clarity and practical advice. The conversation centered on several key themes:

*   **Architectural Best Practices:** Users highlighted the paper's core insight that `io_uring` is not a drop-in replacement and requires a thread-per-core architecture that overlaps I/O and computation. This was seen as a crucial takeaway.
*   **Advanced Features and Trade-offs:** A detailed exchange clarified the nuances of polling. Users learned that `IOPOLL` (for storage) and `SQPOLL` (for submission) are not mutually exclusive, but `IOPOLL` requires separate rings for storage and network I/O. The trade-off between CPU usage and latency was also discussed.
*   **Deployment and Security Concerns:** A significant point was the practical limitation of `io_uring` in containerized and cloud environments. It was noted that Docker and many cloud sandboxes block `io_uring` by default due to its large attack surface. However, it was clarified that this restriction typically does not apply to VMs where users have more control over the kernel.
*   **Technical Corrections and Clarifications:** The community engaged in constructive critique. One user corrected a paper's statement that enterprise SSDs don't require `fsync`, explaining that while drive-level power-loss protection exists, software must still issue `fsync` to guarantee data durability. Another user pointed out a minor error (confusing `malloc` with a syscall), which the authors acknowledged.
*   **Implementation and Tooling:** A user inquired about library support for advanced features like NVMe passthrough, noting that while `liburing` supports it, other language-specific libraries (like Zig's standard library or some Rust crates) may lag behind.
