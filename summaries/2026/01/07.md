## [High-Performance DBMSs with io_uring: When and How to use it](https://arxiv.org/abs/2512.04859)
**Score:** 138 | **Comments:** 36 | **ID:** 46517319

> **Article:** The paper "High-Performance DBMSs with io_uring: When and How to use it" provides a practical guide for database management system (DBMS) developers on leveraging the Linux `io_uring` interface for significant performance gains. The authors argue that `io_uring` is not a simple drop-in replacement for traditional I/O calls but requires a fundamental architectural shift. Key recommendations include moving from a worker-thread model to a "ring-per-thread" design that overlaps computation and I/O within the same thread to maximize efficiency. The paper outlines a step-by-step approach: first, confirm I/O is the bottleneck; second, refactor the architecture for asynchronous batching; and only then, adopt advanced `io_uring` features like fixed buffers, zero-copy, and polling. It details the performance benefits and trade-offs of specific features, such as `IOPOLL` for ultra-low latency storage I/O (at the cost of disabling non-polled I/O like sockets on the same ring) and `SQPOLL` for reducing submission overhead. The research demonstrates that a properly implemented `io_uring` architecture can dramatically increase transaction throughput, especially on modern hardware like NVMe SSDs and high-speed NICs.
>
> **Discussion:** The Hacker News discussion was overwhelmingly positive, with users praising the paper for its clarity and practical advice. The authors were actively engaged, answering questions and clarifying points.

A key theme was the practical application and limitations of `io_uring`. Users sought clarification on advanced configurations, such as whether to use separate rings for storage (with `IOPOLL`) and network I/O on the same thread, to which the authors confirmed this is necessary. The trade-offs between polling modes (`IOPOLL` vs. `SQPOLL`) were also discussed, with the authors explaining that `SQPOLL` reduces submission overhead while `IOPOLL` eliminates interrupt latency for storage, and they can be used together. The consensus was that a thread-per-core architecture is the ideal model for modern DBMSs using `io_uring`.

Deployment and environment constraints were a significant point of discussion. Multiple users raised concerns about `io_uring` availability in containerized and cloud environments. It was clarified that while Docker and some cloud sandboxes block `io_uring` by default due to security risks, it is fully available on virtual machines from major cloud providers, making it viable for many production use cases.

Finally, the discussion highlighted several technical nuances and corrections. A notable debate emerged around the paper's claim that enterprise SSDs might not require `fsync` for durability, with commenters strongly refuting this, clarifying that while drive-level capacitors protect the drive's own buffer, the application must still issue `fsync` to guarantee data is handed over from the OS. Other points included a correction that `malloc` is not a syscall, and a practical note that many user-space libraries lag behind `liburing` in supporting the latest `io_uring` features, potentially requiring developers to use FFI bindings.
