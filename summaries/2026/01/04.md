*   **Niche Tooling:** A significant pain point remains for users of specific automation and scripting tools (e.g., autoclickers, screen recording), which are more complex to implement under Wayland's security model.

In essence, the community is split between those who see Wayland as a flawed but necessary and now-functional evolution, and those who view it as a regression in design and stability that offers no compelling benefit for their established X11 workflows.

## [Maybe comments should explain 'what' (2017)](https://www.hillelwayne.com/post/what-comments/)
**Score:** 174 | **Comments:** 155 | **ID:** 46486780

> **Article:** The 2017 article by Hillel Wayne argues against the dogmatic belief that all "what" comments (which explain what the code is doing) are bad and should be replaced by code structure. He contends that while "why" comments (explaining the reason for the code) are valuable, there are legitimate cases where "what" comments are necessary. These cases include explaining complex algorithmic logic where variable names would be too verbose, documenting non-obvious performance optimizations, or clarifying intricate pointer manipulations. Wayne suggests that forcing such logic into self-documenting code can sometimes make it less readable and harder to follow than the original code with a clarifying comment.
>
> **Discussion:** The Hacker News discussion largely agrees with the article's premise, with many developers expressing frustration with overly dogmatic refactoring styles often attributed to "Uncle Bob" (Robert C. Martin). A common theme is the criticism of excessive method extraction, which can hinder readability by forcing developers to constantly "jump to definition" and lose context.

However, the conversation also refines the article's point. Several commenters suggest that the distinction between "what" and "why" is often blurred, especially when code implements complex business rules. In such cases, a comment explaining the business rule serves as both a "what" (what the code is enforcing) and a "why" (the business reason for it). The consensus is pragmatic: comments should be used to aid understanding, regardless of a strict "what" vs. "why" classification. The goal is clarity for the next developer, and if that requires a "what" comment to explain a tricky piece of logic, it should be added.
