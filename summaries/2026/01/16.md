## [JuiceFS is a distributed POSIX file system built on top of Redis and S3](https://github.com/juicedata/juicefs)
**Score:** 153 | **Comments:** 91 | **ID:** 46637165

> **Article:** The article links to the GitHub repository for JuiceFS, a distributed POSIX file system designed for cloud storage. JuiceFS decouples data and metadata: it stores data objects in an object store (like Amazon S3) and manages metadata in a database (like Redis). This architecture aims to provide the performance of a local file system with the scalability and durability of cloud object storage, suitable for large-scale data processing, AI workloads, and shared storage across multiple machines.
>
> **Discussion:** The Hacker News discussion offers a nuanced view of JuiceFS, balancing its utility with significant architectural trade-offs. A central theme is the comparison with alternatives, particularly ZeroFS, which was promoted in the comments for its performance on small file workloads and simpler deployment (requiring only S3, no separate database). However, this comparison was met with skepticism regarding its methodology and fairness, with some users questioning the validity of the benchmark results.

Several commenters shared practical experiences and concerns. A key drawback highlighted is JuiceFS's dependency on a metadata store (like Redis), which becomes a critical bottleneck and single point of failure; if the metadata is lost, the data in object storage becomes inaccessible due to JuiceFS's proprietary format. This sparked a debate about the reliability of using Redis for such a critical function and a desire for more robust metadata backends. Conversely, the team behind "Sprites" shared their positive experience using a heavily modified JuiceFS with a custom SQLite metadata backend, demonstrating its adaptability for specific high-performance needs.

The suitability of JuiceFS as an NFS replacement was also debated. While it addresses some of NFS's well-known issues (like locking and stale file handles), concerns were raised about the performance overhead and limitations of FUSE (Filesystem in Userspace), which JuiceFS relies on. Ultimately, the discussion concluded that while JuiceFS is a powerful tool for scaling storage and is used in production by many, it requires careful consideration of its metadata architecture and may not be a universal solution for all POSIX workloads, especially those requiring absolute POSIX semantic compliance.
