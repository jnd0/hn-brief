## [What happened to WebAssembly](https://emnudge.dev/blog/what-happened-to-webassembly/)
**Score:** 323 | **Comments:** 306 | **ID:** 46551044

> **Article:** The article "What happened to WebAssembly" addresses the perception that WASM has failed to live up to the initial hype of fundamentally replacing JavaScript and the web ecosystem. The author argues that this perception stems from unrealistic expectations rather than a lack of progress. The article clarifies that WASM's primary design goal was not to be a standalone application platform that bypasses the web, but rather to be a compilation target for performance-critical code to run alongside JavaScript. It highlights key limitations that prevent WASM from being a full replacement, such as the lack of direct DOM access (requiring JS glue code), the overhead of data marshaling between WASM and JS, and the large binary sizes that impact download times. Ultimately, the article concludes that WASM is succeeding in its intended niche: accelerating complex applications like Figma, AutoCAD, and Photoshop in the browser, and enabling new use cases like running existing desktop/server applications on the web, rather than being a "JavaScript killer."
>
> **Discussion:** The Hacker News discussion reveals a community grappling with the gap between WebAssembly's immense potential and its current, more specialized reality. A central theme is the disappointment felt by those who envisioned WASM as a "write once, run anywhere" holy grail that would replace not just JavaScript, but also traditional server containers and desktop applications. This grand vision makes its current use in performance hotspots feel like a letdown.

However, many commenters push back against this narrative, arguing that the goal was never to replace JavaScript entirely. They point out that the modern JS ecosystem (React, TypeScript, Vite) is so productive and performant for most tasks that a WASM replacement is unnecessary. For the few cases where extreme performance is needed, WASM is already a success story (e.g., image editors, Figma).

Practical challenges remain significant hurdles to wider adoption:
*   **Performance Trade-offs:** While compute is fast, the large binary sizes are a major issue for users on slow connections, and the overhead of communicating between WASM and JavaScript can negate performance gains.
*   **Ecosystem Maturity:** The tooling and documentation are still considered far behind JavaScript's, making development more difficult for the average developer.
*   **Governance and Standards:** There is concern about infighting and political disagreements, particularly around the WASI standard, which could slow down progress.

Finally, a counter-narrative emerged: WASM is already deeply integrated and essential to the modern web, but invisibly. Many popular build tools and even parts of frameworks like React are written in Rust and compiled to WASM, meaning that "pulling the plug" on it would break a huge portion of the frontend development workflow. The consensus is that WASM is a powerful tool for power users and specific, demanding applications, but not a general-purpose replacement for the web's core technologies.
