## [Make.ts](https://matklad.github.io/2026/01/27/make-ts.html)

> **Article:** The article "Make.ts" by matklad proposes using TypeScript (via Deno) as a replacement for shell scripts and Makefiles for ad-hoc automation tasks. The core idea is to create a `make.ts` file in project directories where developers can persist useful command snippets they would otherwise type in the shell or store in shell history. The author emphasizes this is for capturing interactive, exploratory commands—not writing "proper" scripts—using Deno's built-in shell templating (`$` tagged template literals) for easy command execution. The approach leverages TypeScript's type safety, IDE support, and cross-platform compatibility while avoiding Make's syntax quirks and shell scripting's fragility.
>
> **Discussion:** The discussion reveals strong polarization around shell scripting alternatives, with several distinct camps emerging. Many developers share the author's frustration with Bash as a scripting language—particularly around conditionals, string manipulation, and cross-platform inconsistencies—leading to enthusiastic endorsement of TypeScript/Deno or Python as replacements. Deno specifically garners praise for its ability to import via relative file paths and embed dependencies directly in scripts, though some find its async/await syntax and JavaScript warts (loose equality, number precision, prototype system) off-putting.

Skeptics question whether this approach adds meaningful value over simple `.sh` files, noting the additional tooling dependency and overhead. Several commenters advocate for established alternatives: `make` itself (for ubiquity and autocomplete), `just` (as a simpler Make replacement), `mise` (as a unified toolchain manager with task running), or even `fzf` with shell history search as a lighter solution for command reuse. A recurring tension surfaces between "personal productivity scripts" and "shareable project infrastructure." Some developers explicitly avoid versioning ad-hoc scripts due to hardcoded paths, local assumptions, and maintenance burden; others argue that capturing this contextual knowledge benefits teams.
