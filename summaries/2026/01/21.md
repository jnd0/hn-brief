> **Article:** The article "Unconventional PostgreSQL Optimizations" by Haki Benita explores advanced techniques to solve a common database problem: efficiently enforcing uniqueness on long text columns without incurring the high storage costs of a standard B-tree index on the column itself. The author presents two primary solutions. The first is using a partial B-tree index on a truncated version of the column (e.g., `CREATE UNIQUE INDEX ON table (left(url, 32))`), which is fast but can lead to hash collisions and isn't perfectly unique. The second, more robust solution involves creating a hash index on the full column (`CREATE UNIQUE INDEX ON table USING HASH (url)`). This method is highly space-efficient, as hash indexes store only the hash value, not the original data, making them ideal for enforcing uniqueness on large values. The article also touches on using generated columns and the `MERGE` command as alternative strategies, providing a comprehensive look at solving this specific performance bottleneck.
>

The technical comments focused on a few main points:
*   **Hash Indexes vs. Generated Columns:** A sub-thread debated the trade-offs between using a hash index (as recommended in the article) versus a stored generated column. The consensus was that while generated columns are convenient, they consume extra storage by materializing the result, which defeats the primary goal of the article's optimization.
*   **PostgreSQL's Query Planning:** One user critiqued PostgreSQL's lack of plan caching for simple queries, contrasting it with other databases. This sparked a detailed reply explaining that PostgreSQL *does* reuse plans for prepared statements, but the per-connection cache and the generic nature of the plan can sometimes lead to suboptimal performance, especially with complex parameters like arrays.

## [Linux kernel framework for PCIe device emulation, in userspace](https://github.com/cakehonolulu/pciem)
**Score:** 211 | **Comments:** 76 | **ID:** 46689065

> **Article:** The article introduces "pciem," a Linux kernel framework that allows for the emulation of PCIe devices directly within userspace. This is achieved by reserving a portion of the host's RAM to act as the device's Base Address Registers (BARs). A userspace program can then interact with this memory, effectively creating a virtual PCIe device that appears on the host's actual PCI bus. The primary goal is to accelerate the development and testing of PCIe drivers and hardware protocols without requiring physical hardware, enabling rapid iteration by simply restarting the userspace process to re-create the device.
>
> **Discussion:** The discussion was highly positive, with users immediately identifying practical applications for the framework. The main themes were driver development, hardware prototyping, and potential hardware offloading use cases.

Several developers highlighted the utility for driver development. It allows for testing driver code against simulated hardware, which is invaluable for debugging complex issues like timing-specific bugs or malformed device responses. This capability was compared to existing tools like `vhci-hcd` for USB, which are widely used for testing drivers in CI environments. The project's userspace-centric design was praised for enabling rapid iteration, as developers can quickly tear down and re-create the virtual device without system reboots.

A key point of discussion was how PCIem differs from existing solutions like QEMU or `libvfio-user`. The author clarified that while those tools are excellent for exposing emulated devices to virtual machines, PCIem operates at a lower level by exposing the device directly to the host's kernel. This allows real, unmodified host drivers to interact with the userspace emulation, a crucial distinction for developing and testing drivers on the host machine itself.

Beyond software development, users explored potential hardware applications. Ideas included using a physical card (e.g., with a Raspberry Pi or a capable MCU) to emulate a PCIe device, enabling tasks like network traffic offloading (e.g., running a VPN on the card) or presenting a ZFS-managed storage array to a host that doesn't natively support it. While some noted the performance limitations of such setups, the concept was seen as a powerful and flexible approach.

Finally, there were practical questions about implementation, such as the need to reserve RAM via a kernel command-line argument (confirmed by the author) and plans for upstreaming the kernel-side support. The project was also noted as a Linux equivalent of Microsoft's older Device Simulation Framework.
