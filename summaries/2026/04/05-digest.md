Lisette, a Rust-inspired language compiling to Go, aims to bridge safety and flexibility. Lisette is a new little language that takes inspiration from Rust’s type system and compiles directly to Go code, aiming to give Go developers stronger safety guarantees without abandoning the Go runtime. The project’s website showcases a syntax that mixes Rust-style enums and pattern matching with Go-like naming conventions, and the author provides a compiler that translates Lisette source into idiomatic Go. The language is positioned as a bridge for existing Go codebases that want more expressive types, such as Option and Result, while still targeting Go’s ecosystem and tooling. Discussion: Commenters praised Lisette’s attempt to bring Rust-style safety to Go, noting that its Option/Result handling could reduce nil-related bugs, but many flagged practical friction points, such as the double-Option awkwardness when interfacing with pure Go code and the continued need for defer-based resource cleanup. A recurring theme was the “why not just use Rust?” argument; participants argued that Go’s garbage-collected, fast-compile workflow still attracts many developers, and Lisette could serve as a low-barrier way to adopt stronger typing. Technical critiques emerged around Go’s multiple-return convention—some warned that treating (T, error) as a simple Result ignores cases like io.Reader’s EOF semantics, prompting the author to acknowledge the issue and open an issue tracker. Opinions diverged on ergonomics versus performance: some saw Lisette as a worthwhile experiment for correctness without sacrificing Go’s runtime, while others worried it adds another language layer that may hinder adoption and complicate existing codebases. Overall, the community expressed cautious optimism, requesting clearer onboarding docs and more concrete examples to gauge real-world utility.
