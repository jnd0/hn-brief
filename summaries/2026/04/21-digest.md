The article discusses "megamerges" in Jujutsu (jj), a modern version control system written in Rust that uses Git as a backend. Megamerges allow developers to combine multiple parallel branches into a single merge operation, enabling workflows where developers work on several unrelated changes simultaneously, then merge them all at once. The author Isaac Corbrey presents this as a productive approach, highlighting jj's superior conflict resolution and painless rebasing as key enablers. The discussion reveals a significant divide between those who don't understand the appeal of megamerge workflows and those who passionately advocate for them. Skeptics like b1temy and umvi argue the approach introduces unnecessary complexity, risking hidden dependencies between branches and making history harder to reason about. Proponents including quicksnap and surajrmal counter that the workflow excels when managing 10-30 in-flight changes with varying dependencies that aren't ready to merge individually - they describe running `jj new A B C` to compose their working environment from multiple changes. A notable critique from SOLAR_FIELDS suggests such complex workflows indicate engineering cultures where PRs aren't reviewed quickly enough. Another thread debates whether jj becomes obsolete in the age of AI-driven development, with uhhhd arguing LLMs abstract away version control concerns entirely, while others push back that agents still struggle with complex merge conflicts. Several users recommend trying jj since it uses Git as a backend, making the switch low-risk, while others note the learning curve involves moving beyond Git muscle memory.
