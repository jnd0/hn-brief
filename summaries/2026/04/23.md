## [Show HN: Honker – Postgres NOTIFY/LISTEN Semantics for SQLite](https://github.com/russellromney/honker)
**Score:** 196 | **Comments:** 26 | **ID:** 47874647

> **Project:** Honker is a project that adds Postgres-style NOTIFY/LISTEN semantics to SQLite, enabling push-style event delivery across processes with single-digit millisecond latency. Rather than using a daemon broker, Honker polls the SQLite WAL (Write-Ahead Log) file using lightweight stat(2) syscalls to detect changes. The project includes ephemeral pub/sub, durable work queues with retries and dead-letter handling, and event streams with per-consumer offsets—all stored as rows in the existing SQLite database and capable of atomic commits with business data, so rollbacks affect both the queue and the transaction.
>
> **Discussion:** The discussion centered on the technical approach of polling the WAL file, with several commenters noting that SQLite's `PRAGMA data_version` or the C API's `SQLITE_FCNTL_DATA_VERSION` might be more appropriate than raw stat(2) calls. Performance was debated—while one user measured stat() at under 1 microsecond per call, others cautioned that syscall overhead extends beyond CPU time to include cache line locking and L3 memory pressure. A key use case distinction emerged: commenters suggested this is most valuable for languages with process-based concurrency (Python, JS, Ruby) rather than languages with native thread support where in-process queues suffice. Several users flagged WAL checkpoint truncation as a potential edge case where events could be missed, and questions arose about compatibility with Litestream and SQLAlchemy. The atomic commit capability—where queue entries roll back alongside business data—was widely cited as Honker's main advantage over external IPC solutions.
