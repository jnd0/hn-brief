## [Show HN: Honker – Postgres NOTIFY/LISTEN Semantics for SQLite](https://github.com/russellromney/honker)
**Score:** 256 | **Comments:** 62 | **ID:** 47874647

> **Project:** Honker is a tool that adds Postgres-style NOTIFY/LISTEN semantics to SQLite, enabling push-style event delivery with single-digit millisecond latency. Instead of running a daemon broker, it monitors the SQLite WAL (Write-Ahead Log) file using lightweight stat(2) system calls to detect changes. The project provides three primitives built as rows in the existing database: ephemeral pub/sub (like pg_notify), durable work queues with retries and dead-letter support (like pg-boss or Oban), and event streams with per-consumer offsets. All operations can commit atomically with business writes, so rollbacks drop both the notification and the data.
>
> **Discussion:** The discussion explored whether this tool is mainly useful for languages with process-based concurrency (Python, JS, Ruby) versus threaded languages where in-process coordination is simpler. Several commenters suggested alternatives to stat(2) polling, including PRAGMA data_version and SQLITE_FCNTL_DATA_VERSION, with the author eventually planning a hybrid approach combining high-frequency data_version checks with lower-frequency stat() calls. Performance concerns were largely dismissed—syscalls are fast enough (under 1μs per call), though one commenter noted hidden costs like CPU cache contention. Edge cases were discussed extensively: clock drift, inode changes, WAL checkpoint truncation, and the difficulty of detecting file deletions when using fstat. Cross-platform limitations were highlighted, particularly that macOS Darwin silently drops same-process file notifications, making inotify/kqueue unreliable compared to stat polling. Scaling to 10k+ concurrent listeners was identified as a potential problem due to thundering herd issues at the stat() call.
