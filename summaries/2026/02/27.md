## [Show HN: Terminal Phone – E2EE Walkie Talkie from the Command Line](https://gitlab.com/here_forawhile/terminalphone)
**Score:** 301 | **Comments:** 73 | **ID:** 47164270

> **Project:** Terminal Phone is a terminal-based, end-to-end encrypted (E2EE) walkie-talkie application designed for use over the Tor network. It uses a v3 onion address as both the cryptographic identity and the NAT traversal mechanism, eliminating the need for STUN/TURN servers. The project features real-time audio transmission using the Opus codec, configurable from 6kbps to 64kbps, with a notable real-world latency of approximately 2-3 seconds due to Tor's characteristics. The application is implemented as a shell script, which introduces limitations like base64 encoding for binary data (adding ~30% overhead) and requires separate tools like ffmpeg for audio processing on Termux. It supports one-on-one communication and plans for group calls, with encryption handled via OpenSSL ciphers, though the project acknowledges the redundancy of this layer given Tor's inherent E2EE properties.
>
> **Discussion:** The discussion centered on the architectural and practical trade-offs of Terminal Phone. Key themes included the 2-3 second latency for ~20KB Opus chunks over Tor, with users debating whether this is acceptable for real-time communication versus store-and-forward models. Technical insights emerged around the choice of 21 OpenSSL ciphers, with some users criticizing the redundancy and advocating for a single default cipher to avoid "cipher fixation" vulnerabilities, while others argued for cryptographic flexibility. The shell script implementation and base64 encoding were discussed as necessary compromises due to bash limitations, though users noted the overhead and workflow complexities on platforms like Termux. The project's use of Tor for identity and NAT traversal was praised for its simplicity but questioned regarding its effectiveness against sophisticated surveillance. Community reactions highlighted both appreciation for the innovative approach and suggestions for improvements, such as reducing latency through one-hop Tor circuits or exploring alternative protocols like Veilid for better real-time performance.
